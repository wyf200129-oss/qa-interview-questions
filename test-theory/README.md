## Q1. 偶发性问题（数据不同步/单据异常）如何先于业务发现并预防？

### 问题描述

系统存在偶发性问题，比如模块间数据不同步、出库单生成凭证偶尔失败、单据作废后回滚不彻底。这种问题不是每次必现，但一旦发生就会导致业务对账出错，影响财务数据准确性。怎么做到在业务人员发现之前就拦截住？

### 核心思路

**针对这类模块数据不同步、偶发的数据异常问题，核心策略是搭建接口自动化框架，把销售改价、出库生成凭证、单据作废回滚这类高频联动场景全部固化成用例。后续每次迭代、版本上线都执行自动化回归，哪怕是偶发性、不易复现的问题，系统也能提前拦截，做到先于业务人员发现问题，避免单据对账出错后再反馈，减少线上业务损失。**

---

### 一、第一步：梳理高频联动场景，固化用例

先盘点业务中哪些场景最容易出现数据不同步：

| 业务场景 | 涉及模块 | 数据不同步风险 |
|:---:|:---:|:---:|
| 销售改价 | 销售 → 库存 → 财务 | 改了销售价但库存成本未同步，毛利率算错 |
| 出库生成凭证 | 仓储 → 财务 ERP | 出库单生成了但凭证未生成，月底对不上账 |
| 单据作废回滚 | 全链路 | 作废了销售单但库存没加回来，库存虚减 |
| 采购入库 → 库存更新 | 采购 → 仓储 | 入库了但库存量没加，下游缺货 |
| 调拨单审核 → 双仓更新 | 仓储A → 仓储B | A仓扣了但B仓没加，两边库存都不对 |

然后针对每个场景，用 Python + Requests 封装成自动化用例。

**示例：出库生成凭证场景**

```python
class TestOutboundToVoucher:
    """出库单生成凭证 — 自动化回归用例"""

    def test_outbound_create_voucher(self, api_client):
        """正常流程：出库 → 自动生成凭证"""
        # Step 1: 创建出库单
        outbound_resp = api_client.post("/wms/outbound/create", json={
            "sku_code": "SKU001",
            "quantity": 100,
            "warehouse_id": "WH001"
        })
        assert outbound_resp.status_code == 200
        outbound_id = outbound_resp.json()["id"]

        # Step 2: 审核出库单（触发凭证生成）
        audit_resp = api_client.post(f"/wms/outbound/{outbound_id}/audit")
        assert audit_resp.status_code == 200

        # Step 3: 验证凭证是否生成
        voucher_resp = api_client.get(f"/finance/voucher/query", params={
            "biz_type": "outbound",
            "biz_id": outbound_id
        })
        assert voucher_resp.status_code == 200
        assert voucher_resp.json()["total"] > 0, "出库单未生成凭证！数据不同步！"
        assert voucher_resp.json()["items"][0]["amount"] == 100 * 10.0  # 数量×单价

    def test_outbound_voucher_consistency(self, api_client):
        """对账：出库单总金额 vs 凭证总金额"""
        # 查当天的所有出库单
        outbounds = api_client.get("/wms/outbound/list", params={
            "date": "2026-06-21"
        }).json()["items"]
        outbound_total = sum(o["amount"] for o in outbounds)

        # 查出库生成的凭证
        vouchers = api_client.get("/finance/voucher/list", params={
            "biz_type": "outbound",
            "date": "2026-06-21"
        }).json()["items"]
        voucher_total = sum(v["amount"] for v in vouchers)

        # 对账
        assert abs(outbound_total - voucher_total) < 0.01, \
            f"对账不平！出库:{outbound_total} 凭证:{voucher_total}"
```

## Q2. 非常罕见的偶发性 Bug（跑100次都不触发一次）怎么处理？

### 问题描述

有些 Bug 比普通偶发问题更难搞——回归跑了几十次都是绿的，线上一个月才出现一两次，日志不完整、复现条件极其苛刻。这种"偶发中的偶发"怎么定位和预防？

### 答案

这种极稀有的偶发 Bug（业内也叫 Heisenbug / 海森堡 Bug）是最难啃的硬骨头。常规的自动化回归很难抓，因为概率太低。需要从 **被动防御 → 主动追踪 → 线索拼图 → 事后兜底** 四个维度来处理。

---

### 一、被动防御：灰度发布 + 监控兜底

既然自动化回归抓不到，那就靠"有限的灰度范围 + 全面的监控覆盖"来兜住它。

**1. 灰度发布策略**

全量发布的风险太高 → Bug 一上线就是全网故障。改成灰度发布 → Bug 只影响 5% 的用户 → 监控发现后立即回滚 → 影响范围最小。

```python
class GrayRelease:
    """按用户 ID 取模做灰度"""
    def __init__(self, gray_percent=5):
        self.gray_percent = gray_percent

    def is_in_gray(self, user_id):
        return hash(str(user_id)) % 100 < self.gray_percent
```

**2. 全链路业务埋点**

针对这种极稀有 Bug，普通日志粒度不够——出现时只有零散信息，拼不出全貌。因此要在关键业务操作增加专门的全链路追踪日志，和业务日志分开存储：

```python
class BizTracer:
    """业务操作全链路追踪器"""

    def __init__(self, biz_type, biz_id):
        self.trace_id = str(uuid.uuid4())[:8]
        self.biz_type = biz_type
        self.biz_id = biz_id
        self.steps = []

    def step(self, action, status="ok", detail=None):
        self.steps.append({
            "trace_id": self.trace_id,
            "biz_type": self.biz_type,
            "biz_id": self.biz_id,
            "action": action,
            "status": status,
            "detail": detail,
            "timestamp": datetime.now().isoformat()
        })

    def finish(self, final_status):
        self.step("__finish__", final_status)
        logger.info(json.dumps({
            "trace_id": self.trace_id,
            "steps": self.steps,
            "final_status": final_status
        }))


# 在出库生成凭证的代码中埋点
def create_voucher_from_outbound(outbound_id):
    tracer = BizTracer("outbound_voucher", outbound_id)
    try:
        tracer.step("查询出库单")
        outbound = get_outbound(outbound_id)
        tracer.step("查询出库单完成", detail=f"金额={outbound.amount}")

        tracer.step("生成凭证")
        voucher = voucher_service.create(outbound)
        tracer.step("凭证生成完成", detail=f"凭证ID={voucher.id}")

        tracer.finish("success")
    except Exception as e:
        tracer.step("异常", "error", detail=str(e))
        tracer.finish("failed")
        raise
```

这样当月度的"凭证生成失败"再次出现时，搜 `biz_type=outbound_voucher` + `final_status=failed`，就能拿到完整的步骤日志。

**3. 连续失败才告警——避免单次抖动骚扰**

```python
class ConsecutiveFailureAlert:
    def __init__(self, biz_key, threshold=3, window_minutes=60):
        self.redis_key = f"alert:{biz_key}"
        self.threshold = threshold
        self.window = window_minutes * 60

    def record_failure(self):
        count = redis.incr(self.redis_key)
        if count == 1:
            redis.expire(self.redis_key, self.window)
        if count >= self.threshold:
            send_alert(f"【{self.biz_key}】连续 {count} 次失败")
            redis.delete(self.redis_key)
```

```python
target = datetime.now().replace(hour=5, minute=0, second=0, microsecond=0)
```

**如果还是复现不了——先预防再继续追**：

有些 Bug 一个月才出现一次，硬等不现实。先加防御代码：

```python
def create_voucher_safe(outbound):
    """带防御的凭证生成——失败后重试一次"""
    try:
        return create_voucher(outbound)
    except VoucherCreateException:
        time.sleep(1)
        try:
            return create_voucher(outbound)
        except VoucherCreateException:
            # 标记为"待人工处理"，不让业务卡住
            mark_for_manual_review(outbound.id)
            return None
```

同时持续追踪，等下次复现时用增强后的日志定位。

### 四、事后兜底：补偿机制 + 数据对账

不管怎么防，极稀有 Bug 总有漏网之鱼。最后一层防线是"出了问题能自动修"。

**每小时自动对账 + 自动修复**：

```python
def auto_reconcile_and_repair():
    """自动对账 + 自动修复"""

    # 查出库已审核但无凭证的单据
    orphan_outbounds = db.query("""
        SELECT o.id, o.amount FROM wms_outbound o
        LEFT JOIN finance_voucher v ON v.biz_type='outbound' AND v.biz_id=o.id
        WHERE o.status='audited' AND v.id IS NULL
          AND o.audited_at > NOW() - INTERVAL 24 HOUR
    """)
    for ob in orphan_outbounds:
        try:
            voucher = voucher_service.create_from_outbound(ob.id)
            notify(f"⛑ 自动修复: 出库单 {ob.id} 补生成凭证 {voucher.id}")
        except Exception as e:
            notify(f"⚠️ 需人工: 出库单 {ob.id} 无法补凭证: {e}")

    # 查作废未回滚库存的单据
    stale_cancels = db.query("""
        SELECT s.id, s.sku_code, s.quantity FROM sale_order s
        WHERE s.status='canceled'
          AND s.canceled_at > NOW() - INTERVAL 24 HOUR
          AND NOT EXISTS (
            SELECT 1 FROM wms_stock_log l
            WHERE l.biz_type='sale_cancel' AND l.biz_id=s.id
          )
    """)
    for sc in stale_cancels:
        try:
            rollback_stock(sc.sku_code, sc.quantity)
            notify(f"⛑ 自动修复: 销售单 {sc.id} 库存已补回滚")
        except Exception as e:
            notify(f"⚠️ 需人工: 销售单 {sc.id} 库存回滚失败: {e}")
```

### 五、面试回答模板

> **"极稀有的偶发 Bug，我分四步处理：**

> **第一，防御兜底**——灰度发布控制影响范围 + 全链路业务埋点（每个关键操作打 trace_id），这样出了问题有完整的操作日志可查。再加连续失败告警，3 次连续失败才通知，避免单次抖动骚扰。

> **第四，自动补偿**——即使没完全消灭，也要有补救机制。每小时跑一次对账巡检，发现出库已审核但无凭证的单据就自动补生成，发现作废未回滚库存就自动回滚，把损失降到最低。"**

## Q3. 平常的测试流程是怎样的？

### 阶段三：测试用例设计

**时间**：测试计划评审通过后

**设计方法**以进销存为例：

**1. 等价类 + 边界值**（最常用）

```
场景：出库单审核

等价类划分：
  有效：出库数量 > 0 且 ≤ 库存量
  无效1：出库数量 ≤ 0
  无效2：出库数量 > 库存量
  无效3：单据状态不是"待审核"

边界值：
  库存量 = 100 时，测 0, 1, 99, 100, 101
```

**2. 场景法（正向 + 异常）**

```
正向场景：
  创建出库单 → 审核 → 生成凭证 → 库存扣减 → 完成

异常场景：
  创建出库单 → 审核（库存不足）→ 审核不通过 → 重新修改 → 再审核
  创建出库单 → 审核通过 → 生成凭证失败 → 人工补凭证
  创建出库单 → 审核通过 → 发现数据错误 → 作废单据 → 库存回滚
```

**3. 正交实验法（多条件组合）**

```
场景：PC端 + App端 销售改价

条件：
  设备类型：PC端 / App端
  改价方式：直接改价 / 批量改价 / 促销改价
  商品类型：正常商品 / 促销品 / 赠品
  用户角色：管理员 / 普通员工

用正交表选 9 组用例覆盖所有组合（不用写 3×3×3×3=81 条）
```

**4. 错误推测法（靠经验）**

```
根据之前踩过的坑，重点覆盖：
  □ 前端点了"提交"后快速连点两次 → 会不会重复提交？
  □ 网络断开后恢复 → 数据会不会丢失？
  □ 凌晨日切任务执行时操作 → 会不会锁冲突？
  □ 超大数据量（10万+ SKU）→ 页面卡不卡？
```

**产出物**：测试用例（禅道管理）

```
模块：进销存-销售改价
  TC001: 正常改价（单商品，价格调高）
  TC002: 正常改价（单商品，价格调低）
  TC003: 批量改价（选择 10 个商品）
  TC004: 改价后查看库存成本是否同步更新
  TC005: 改价 + 并发出库
  TC006: 改价为负数（异常）
  TC007: App端改价
  ...
```

### 阶段五：测试执行

**时间**：开发提测后

**执行顺序**：

```
第一轮：冒烟测试（核心流程）
  目标：确认版本可测，不阻塞
  范围：登录、创建订单、出库、审核等 20 条核心用例
  通过标准：100% 通过
  不通过 → 打回开发修复，不进入详细测试

第二轮：功能测试
  目标：覆盖所有功能点
  范围：全部功能用例（Web端 + App端）
  手段：手工执行 + adb 调试 + Fiddler 抓包辅助定位
  发现 Bug → 提单 → 开发修复 → 验证关闭

第三轮：自动化回归
  目标：防止修改引入新问题
  范围：接口自动化用例（核心链路 90+ 条）
  手段：Jenkins 触发 → pytest + Allure 报告
  通过标准：>= 95% 通过率，无 P0/P1 缺陷

第四轮：专项测试
  兼容性：Chrome / Edge / 微信内置浏览器
  性能：JMeter 单接口压测 + 业务链路压测
  安全：越权、SQL注入、XSS 等基础安全扫描
```

**Bug 生命周期**：

```
发现 → 提交（禅道）→ 开发确认 → 修复 → 测试验证 → 关闭
                                                   ↘ 拒绝/延期 → 沟通确认
```

**Bug 定级**：

| 级别 | 定义 | 处理要求 |
|:---:|:---|:---:|
| P0 | 阻塞主流程，无法继续测试 | 立即修复，2小时内 |
| P1 | 核心功能受影响，有替代方案 | 当天修复 |
| P2 | 非核心功能异常，不影响主流程 | 本迭代修复 |
| P3 | 界面/文案/体验问题 | 排期修复 |

## Q4. 面试中的"讲 Case / 业务演示"环节是什么样的？

### 问题描述

有些面试（尤其是二面/三面）会让候选人当面讲一个自己做过的项目，或者给一个业务场景让你现场设计测试用例。这个环节面试官想考察什么？该怎么准备？

### 答案

这个环节在面试中叫 **Case Presentation** 或 **业务 Demo 环节**，一般出现在二面或三面（技术面 + 交叉面）。面试官不是在听你背书，而是在看你**对业务的理解深度、表达的逻辑性和应对追问的灵活性**。

---

### 一、面试官想考察什么（4 个维度）

| # | 考察点 | 面试官在观察什么 |
|:---:|:---|:---|
| 1 | **业务理解** | 你是机械执行需求还是真的理解了业务逻辑？涉及哪些模块、数据怎么流转、异常怎么处理？ |
| 2 | **测试思维** | 给你一个场景，你能想到哪些 Case？边界、异常、并发、数据一致性——你脑子里有没有这个体系？ |
| 3 | **表达能力** | 能不能在 5 分钟内把一个项目讲清楚？逻辑清晰吗？重点突出吗？ |
| 4 | **应变能力** | 面试官追问时，是卡住还是能顺势接上？修改条件后你能马上调整思路吗？ |

---

### 二、讲 Case 的万能结构化框架

不管是讲项目还是现场设计用例，用这个 **4 步框架** 准没错：

> **背景 → 我做了什么 → 难点与解法 → 效果与收获**

**以你的 ERP 项目为例：**

**Step 1：背景（30 秒）**

> **"我负责的 ERP 进销存系统，是面向服装企业的 Web + App 双端平台，覆盖 13 个业务模块。我主要负责采购、销售、库存三个核心模块的质量保障。"**

**Step 2：我做了什么（1.5 分钟）**

> **"我从 0 到 1 搭建了接口自动化和 UI 自动化两套框架。接口自动化用 Python + Requests + 关键字驱动，实现了单接口多条件覆盖和业务链路自动化，90 条用例接入 Jenkins 每日回归。UI 自动化用 POM 模式，采购/销售高频流程沉淀为可复用用例。"**
>
> **"针对进销存三模块联动场景——比如销售改价后库存成本同步、出库审核后自动生成凭证、单据作废后库存回滚——我把这些高频联动场景全部固化为接口自动化用例，每次版本迭代都跑回归。"**

**Step 3：难点与解法（2 分钟）—— 这里最关键**

> **"最大的难点是偶发的数据不同步问题。比如出库审核后凭证生成偶尔失败，平时测试跑都是绿的，线上一个月出现两三次。自动化回归虽然能拦截一部分，但这种极低概率的抓不到。"**
>
> **"我的解法是三层：**
> **第一，在 CI 中加了对账检查脚本——出库单金额 vs 凭证金额逐笔比对，有差异直接钉钉告警。**
> **第二，做了定时数据一致性巡检——每 30 分钟扫一次'出库已审核但无凭证'的单据，自动补生成。**
> **第三，搭建了全链路业务埋点——每个关键操作打 trace_id，配合连续失败告警（3次连续失败才通知），避免单次抖动骚扰。"**

**Step 4：效果与收获（30 秒）**

> **"效果是：后续几次版本上线，凌晨跑完回归就发现了 3 次凭证未生成、2 次作废回滚不彻底的问题，业务上班前我们就修了。从那以后没再出现过月底对账不平才知道的情况。"**
>
> **"个人收获是：对业务数据流转的理解更深了，也意识到自动化不只是跑用例，对账和补偿机制才是保障数据一致性的最后一关。"**

### 三、现场设计用例的应答模板

面试官给了一个场景让你设计 Case，比如：

> **"设计一个订单出库的测试用例，你打算怎么测？"**

不要上来就一条条列 Case，先用这个结构回答：

**Step 1：先反问确认边界（展示思维严谨）**

```
"我先确认几个边界条件："
□ 出库流程覆盖 PC 端和 App 端吗？
□ 出库单允许多个 SKU 还是一个？
□ 出库后是否触发其他业务动作？（生成凭证？更新库存？）
□ 库存不足时是阻断还是允许负库存出库？
```

**Step 2：再按维度分类回答（展示体系化思维）**

```
正常流程：
  - 创建出库单（1个SKU）→ 审核 → 凭证生成 → 库存扣减 → 完成
  - 创建出库单（多个SKU）→ 审核 → 凭证生成 → 各SKU库存分别扣减

异常流程：
  - 出库数量 > 库存量 → 审核不通过
  - 出库数量 = 0 或负数 → 系统拦截
  - 出库单审核时网络中断 → 数据一致性检查
  - 出库审核后生成凭证失败 → 是否有补偿机制
  - 出库单已审核，又点击审核 → 幂等处理

边界值：
  - 出库数量 = 1（最小）
  - 出库数量 = 库存量（刚好够）
  - 出库数量 = 99999（超大数据）

并发场景：
  - 50 人同时创建出库单
  - 同一 SKU 同时出库 + 入库
  - 出库审核同时，另一个人操作作废

数据一致性校验：
  - 出库后，库存表是否扣减正确
  - 出库单金额 vs 生成凭证的金额是否一致
  - 多仓调拨出库后，两仓库存总和是否不变
```

**Step 3：补充自动化 + 监控覆盖（展示工程能力）**

```
这些 Case 我们都会纳入接口自动化回归，每次版本上线前跑一遍。
同时线上有对账脚本，每 30 分钟扫一次出库单和凭证的差异，自动告警。
```

## Q5. 前后端 Bug 怎么区分？

### 问题描述

测试过程中发现一个 Bug，怎么判断是前端的问题还是后端的问题？面试官问这个问题是想看你**定位问题的能力**，而不是只会提 Bug。

### 答案

前后端 Bug 的区分是测试的基本功，也是面试高频题。核心思路是：**通过抓包工具看接口，接口对了就是前端问题，接口错了就是后端问题。**

---

### 一、核心原则：抓包定位法

不管你面前是什么 Bug，先开 Fiddler / Charles / 浏览器 DevTools 看接口：

```
页面操作 → 抓包看请求
               │
     ├── 请求没发出去？            → 前端问题
     │     （按钮没绑事件、JS报错、参数没拼好）
     │
     ├── 请求发出了，响应不对？     → 后端问题
     │     （返回500、数据字段缺失、状态码不对）
     │
     └── 请求和响应都正常，但页面显示不对？ → 前端问题
           （前端解析数据错了、渲染逻辑有Bug）
```

---

### 二、7 种常见场景的判断方法

#### 场景 1：页面点了按钮没反应

```
操作：点击"提交订单"按钮，页面没任何反应

排查：
  1. F12 → Console 看有没有 JS 报错
     - 有报错 → 前端问题（JS 异常、事件绑定失败）
  2. Network 看有没有请求发出
     - 没请求 → 前端问题（按钮没绑定接口调用）
     - 有请求（200）→ 看后端返回的数据对不对
```

#### 场景 2：页面报 500 错误

```
操作：提交出库审核，页面返回 500

排查：
  F12 → Network → 看请求返回 500 → 后端问题
  把请求参数和返回信息提给后端开发

  进一步定位：
  让开发看服务器日志，500 一般是代码抛了未捕获异常
```

#### 场景 3：页面数据显示不对

```
操作：查看订单列表，金额应该是 100.00，页面上显示 100

排查：
  1. Network 看接口返回的数据
     - 接口返回 {amount: 100} → 前端问题（没做小数点格式化）
       ✅ 提单给前端：金额字段需保留两位小数
     - 接口返回 {amount: 100.00} 但页面显示 100 → 还是前端问题
     - 接口返回 {amount: null} 或 {amount: "100"} 类型不对 → 后端问题
       ✅ 提单给后端：接口需返回 DECIMAL 类型，且不能为 NULL
```

#### 场景 4：某个条件查询结果不对

```
操作：按"已付款"状态查询，本应有 10 条，只显示了 5 条

排查：
  1. F12 → 看接口 URL 参数
     - 参数传了 status=PAID → 后端返回了 5 条 → 后端问题
       （SQL 查询条件不对或数据本身没有 10 条）
     - 参数传了 status=paid（小写）→ 前端拼参问题
       ✅ 前端问题：参数大小写和后端不匹配
```

#### 场景 5：页面样式错乱 / 白屏

```
操作：打开某个页面，布局错乱或白屏

排查：
  1. Console 看报错
     - "xxx is not defined" / "Cannot read property of undefined"
       → 前端问题（数据还没加载完就渲染了 / 接口返回的数据结构变了）
  2. 看接口是否正常返回
     - 接口挂了返回 500 → 后端问题，前端拿不到数据所以白屏
     - 接口正常但前端解析报错 → 前端问题
```

#### 场景 6：某个功能只在特定浏览器出问题

```
操作：Chrome 正常，Edge/微信内置浏览器报错

→ 前端问题（兼容性）
  典型原因：CSS 兼容性、ES6+ 语法未转译、Web API 差异
  提单给前端：注明浏览器版本和报错信息
```

#### 场景 7：App 端和 Web 端表现不一致

```
操作：Web 端出库审核正常，App 端审核报错

排查：
  1. Fiddler 抓 App 端的请求
     - App 端请求参数和 Web 端不一样 → App 端（前端）问题
     - 请求参数一样，App 端报错 Web 端正常 → 后端问题
       （后端可能对不同端有不同处理逻辑）
```

---

### 三、用 Fiddler/Charles 辅助定位

**Fiddler 常用操作**：

```text
1. 打开 Fiddler → 确保 Capturing 开启
2. 操作页面 → 在 Fiddler 中找到对应请求
3. 看 Inspectors 面板：
   - Request：看请求方法、URL、参数、请求体
   - Response：看返回状态码、响应体
4. 修改请求重放（Composer）：
   - 改参数 → 看后端返回是否正常
   - 改返回 → 看前端渲染是否正确
```

**浏览器 DevTools 常用操作**：

```text
F12 → Network → 按 XHR 过滤 → 只看接口请求
  ├── 看 Status：200 正常 / 4xx 客户端 / 5xx 服务端
  ├── 看 Response：后端返回的数据对不对
  ├── 看 Payload：前端传参对不对
  └── 看 Timing：看哪个环节耗时最长

F12 → Console → 看 JS 错误和网络请求日志

F12 → Elements → 看 DOM 结构 vs 接口数据是否一致
```

---

### 四、前后端 Bug 分类速查表

| 现象 | 大概率 | 小概率 | 验证方法 |
|:---|:---|:---|:---|
| 页面无响应，Console 无报错 | 后端接口超时或挂了 | 前端事件绑定失败 | 看 Network 请求是否发出 |
| 返回 500 / 502 / 503 | 后端异常 | 网关/Nginx配置 | 看响应体和服务器日志 |
| 返回 404 | 接口地址不对 | Nginx 路由配置 | 检查 URL 是否正确 |
| 返回 200 但数据不对 | 后端 SQL/逻辑错误 | 前端传参错误 | 对比请求参数和响应数据 |
| 页面显示正常但数据不刷新 | 前端缓存问题 | 后端返回了旧数据 | 强制刷新看接口返回 |
| 某个用户报错其他人正常 | 后端权限/数据权限 | 前端配置差异 | 对比两个用户的请求参数差异 |

---

### 五、面试回答模板

> **"我区分前后端 Bug 的核心方法是看抓包。遇到 Bug 先开 Fiddler 或浏览器 DevTools 看接口：**

> **如果请求没发出去或者参数拼错了，那就是前端问题；如果请求发出去了但返回的数据不对或者报 500，那就是后端问题；如果请求和响应都正常但页面显示不对，还是前端问题（渲染或者数据解析错了）。**

> **举个例子，之前遇到过销售改价后页面金额显示不对。我抓包一看，接口返回的是 {amount: 100.00}，但页面显示 100。明明是后端给了正确数据，前端没做小数位格式化，所以提单给前端。如果接口返回的 amount 就是错的，那就提单给后端。**

> **还有一些辅助判断技巧：特定浏览器出问题一般是前端兼容性，Chrome 正常 App 报错可能是 App 端传参问题。但最核心的还是抓包看数据流——数据在哪一环出了问题，那一环就是 Bug 归属。"**

**关键词**：前后端 Bug 区分 / 抓包定位 / Fiddler / 浏览器 DevTools / 接口校验

## Q6. 测试计划和测试策略的区别是什么？

### 问题描述

面试官问你测试计划和测试策略的区别，这两个概念经常被混用。考的是你**对测试管理的理解层次**。

### 答案

一句话总结：

> **测试策略是"测什么、怎么测、重点在哪"——方向层面的。测试计划是"谁来做、什么时候做、用什么资源做"——执行层面的。**

用盖房子来类比：

| | 测试策略 | 测试计划 |
|:---|:---|:---|
| 类比 | 建筑设计图 | 施工排期表 |
| 关心 | 房子盖成什么样、用什么材料 | 什么时候打地基、谁负责砌墙 |
| 变化频率 | 项目初期定下来，一般不轻易改 | 每周都可以调整 |
| 谁来定 | 测试负责人 + 架构师 | 测试组长 + PM |

---

### 一、测试策略（Test Strategy）

**定义**：从技术层面确定测试的范围、方法、维度和重点。

**包含的内容**：

```
测试策略文档
├── 测试范围：测什么、不测什么
│   ├── 进销存三模块（采购/销售/库存）
│   └── 不测的范围：报表模块（本次未变更）
├── 测试方法
│   ├── 功能测试：手工执行
│   ├── 接口自动化：Pytest + Requests
│   ├── UI 自动化：Selenium + POM
│   ├── 性能测试：JMeter（核心接口压测）
│   └── 兼容性：Chrome + Edge + 微信浏览器
├── 测试重点
│   ├── 核心链路：销售改价 → 成本同步 → 财务校验
│   ├── 数据一致性：出库金额 vs 凭证金额对账
│   └── 并发场景：多人同时操作同一单据
├── 风险分析
│   ├── 第三方支付接口不稳定 → 提前 mock
│   └── 财务数据敏感 → 测试数据脱敏
└── 测试环境要求
    ├── 一套独立测试环境（与开发环境隔离）
    └── 测试数据用生产脱敏数据
```

**特点**：
- 项目初期定下来，一般不轻易改
- 和架构设计同步进行——接口怎么设计，测试就怎么测
- 决定了自动化框架怎么搭、用例怎么设计

**实际案例**：

> 做 ERP 系统时，我们定的测试策略是：**核心链路全覆盖（自动化）+ 数据一致性巡检（对账脚本）+ 偶发问题兜底（定时巡检）**。这个策略从项目开始到最后没变过。

---

### 二、测试计划（Test Plan）

**定义**：从管理层面确定谁在什么时间做什么事。

**包含的内容**（以一个迭代为例）：

```
测试计划（版本 v2.3.1）
├── 时间安排
│   ├── 6月1日-6月3日：需求评审 + 测试要点
│   ├── 6月4日-6月7日：用例设计 + 评审
│   ├── 6月8日-6月14日：功能测试（第1轮）
│   ├── 6月15日-6月18日：自动化回归 + 性能测试
│   ├── 6月19日：产品验收
│   └── 6月20日：上线
├── 人员分工
│   ├── 龙哥：销售 + 库存模块（主责）
│   ├── 小张：采购模块 + 自动化回归
│   └── 小王：性能测试 + 兼容性测试
├── 交付物
│   ├── 测试用例（禅道）
│   ├── 自动化测试报告（Allure）
│   ├── 性能测试报告
│   └── 上线验收 Checklist
├── 准入准出条件
│   ├── 准入：开发提测 + 冒烟通过
│   └── 准出：P0/P1清零 + 自动化≥98%
└── 风险应对
    ├── 开发延期 → 压缩功能测试时间，保证自动化回归
    └── 环境不稳定 → 申请独立测试环境
```

**特点**：
- 每个迭代/版本都有一份
- 随项目进度动态调整
- 和项目管理工具（禅道/Jira）联动

---

### 三、两者对比一览

| 对比维度 | 测试策略 | 测试计划 |
|:---:|:---|:---|
| **层次** | 战略层 | 执行层 |
| **关注** | 测什么、怎么测、重点在哪 | 谁做、何时做、用什么资源 |
| **时间** | 项目初期，基本不变 | 每个迭代都有一份 |
| **受众** | 测试团队 + 架构师 | 全员（测试/开发/PM/产品） |
| **粒度** | 粗（方法和方向） | 细（每天做什么） |
| **输出** | 测试策略文档（一页纸） | 测试计划（禅道/Excel） |
| **变更频率** | 低（需求重大变更才改） | 高（每周调整） |
| **类比** | 打篮球的战术体系 | 这场比赛的轮换阵容 |

---

### 四、实战中的关系

测试策略和测试计划不是二选一，而是配合关系：

```
项目启动
    ↓
定测试策略（测什么、怎么测） ← 这次不改
    ↓
排测试计划（谁做、何时做）   ← 每迭代更新
    ↓
按计划执行 → 发现问题 → 调整计划（策略不变）
    ↓
迭代结束 → 复盘 → 下个迭代更新计划
```

**实际例子**：

> 我们做 ERP 的测试策略一直是"核心链路自动化 + 数据一致性巡检 + 偶发兜底"，从项目开始到最后没变过。但测试计划每两周更新一次——这个迭代测改价功能，下个迭代测出库功能，人员安排和时间点都不一样。

---

### 五、面试回答模板

> **"测试策略和测试计划的区别，我用四句话概括：**
>
> **测试策略是方向——决定了测什么维度、用什么方法、重点在哪、风险和应对。它像建筑设计图，项目初期定下来一般不改。**
>
> **测试计划是执行——决定了谁来做、什么时候做、交付物是什么。它像施工排期表，每个迭代更新一次。**
>
> **举个例子，我做 ERP 时测试策略一直是"核心链路自动化 + 数据一致性巡检 + 偶发兜底"，但这个策略不变的情况下，测试计划每两周排一次——这个迭代测销售改价，下个迭代测出库管理，时间点和人员都不一样。**
>
> **简单说：策略是"想清楚怎么打"，计划是"排好谁上场"。"**

---

**关键词**：测试策略 / 测试计划 / 测试管理 / 测试范围 / 执行计划 / 风险分析

## Q7. 结合 ERP 业务，测试用例设计一般用什么方式比较多？

### 问题描述

面试官问你平时用例设计用什么方法，等价类、边界值、场景法……你在实际工作中哪些用得最多？结合你的 ERP 业务场景来说。

### 答案

做 ERP 进销存测试时，用的最多的是这四种方法，按频率排序：

> **场景法 > 等价类边界值 > 错误推测法 > 正交实验法**

---

### 一、场景法（用得最多，占 50%）

**为什么最多**：ERP 的核心是"业务流"，不是单点功能。一个操作涉及多个模块联动，场景法最适合覆盖这种链路。

**常用场景**：

```text
正向场景（每天测）：
  创建采购单 → 审核 → 入库 → 库存增加 → 生成应付凭证

  创建销售单 → 审核 → 扣库存 → 出库 → 生成应收凭证

  创建调拨单 → 审核 → A仓出库 → B仓入库 → 两仓库存总和不变

异常场景（最容易出 Bug）：
  创建销售单 → 审核 → 库存不足 → 审核不通过 → 修改数量 → 重新审核

  创建销售单 → 审核 → 出库 → 生成凭证失败 → 人工补凭证

  创建销售单 → 审核 → 发现数据错误 → 作废 → 库存回滚 → 凭证冲红

  已审核的出库单，再次点击审核（幂等测试）
```

**场景法的核心思路**：

```
不要只测"单个按钮好不好用"
要测"业务走完一条完整的路，每个环节的数据对不对得上"
```

### 二、等价类 + 边界值（占 30%）

**用在什么地方**：输入框、数量、金额、日期等有明确输入范围的字段。

**ERP 中的常见边界场景**：

```text
数量类：
  出库数量 = 0           → 非法，系统拦截
  出库数量 = 1           → 最小有效值
  出库数量 = 库存量      → 刚好够
  出库数量 = 库存量 + 1  → 库存不足，审核不通过
  出库数量 = 999999      → 超大数据，性能边界

金额类：
  改价 = 0              → 按业务规则判断是否允许
  改价 = 0.01           → 最小金额
  改价 = 负数           → 非法
  改价 = 9999999.99     → 最大金额边界

日期类：
  单据日期 = 当天
  单据日期 = 上个月
  单据日期 = 跨年
  单据日期 = 未来日期    → 是否允许？
```

**实际案例**：

> 有次测试出库数量，我测了 `出库数量 = 库存量` 刚好过，但没测 `出库数量 = 0`。上线后发现用户输入 0 提交了，系统没拦截，生成了空出库单。后来我把边界值加到用例库，再没出现过。

### 三、错误推测法（占 15%）

**为什么 ERP 尤其需要这个**：因为踩过的坑太多，要把历史问题沉淀为测试用例。

**ERP 中常见的"老坑"**：

```text
并发类（最常踩）：
  - 两个用户同时审核同一张单据
  - A 在改价，B 在出库，操作同一个 SKU
  - 用户快速连点两次"提交"

数据一致性类（最致命）：
  - 出库单审核了，凭证没生成
  - 销售单作废了，库存没加回来
  - 调拨单完成了，两个仓的库存总和变了

状态冲突类：
  - 已作废的单据还能审核
  - 已完成的出库单还能修改
  - 已生成凭证的单据还能作废

环境类：
  - 网络断开后恢复 → 数据是否丢失？
  - 凌晨日切时操作 → 会不会锁冲突？
  - 浏览器后退 → 是否会重复提交？
```

**实际案例**：

> 之前线上出过一个问题：用户开了两个 Tab，一个改价一个出库，同时操作同一个 SKU。改价提交后库存成本变了，出库那笔按新成本算的，但财务那笔凭证是按旧成本生成的。月底对账发现差了 5000 块。
>
> 之后我把"改价 + 并发出库"写进用例库，每次回归必跑。

### 四、正交实验法（占 5%）

**用在什么地方**：条件组合多、但不需要全量测试的时候，用正交表减少用例数量。

**ERP 中的例子**：

```text
场景：测试"销售改价"的权限控制

条件（4个维度）：
  用户角色：管理员 / 销售主管 / 普通销售
  改价方式：单商品改价 / 批量改价
  改价幅度：上调 / 下调
  商品类型：正常商品 / 促销品 / 赠品

全量组合：3 × 2 × 2 × 3 = 36 条
正交表选 9 条就能覆盖所有组合
```

### 五、实战中怎么搭配使用

以一个具体的功能为例——**出库审核**：

```text
Step 1：场景法（先画业务链路）
  正向：创建出库单 → 审核 → 库存扣减 → 凭证生成
  异常1：库存不足 → 审核不通过
  异常2：审核后凭证生成失败 → 自动补偿
  异常3：已审核的单据再次审核 → 幂等

Step 2：等价类边界值（填输入细节）
  出库数量：0 / 1 / 库存量 / 库存量+1 / 999999
  SKU数量：1个 / 10个 / 100个

Step 3：错误推测法（补充踩过的坑）
  并发：两人同时审核同一出库单
  状态：已作废的单据能否审核？
  环境：审核时网络断开

Step 4：正交实验法（收尾）
  如果还有多条件组合没覆盖到，用正交表补几条
```

### 六、面试回答模板

> **"做 ERP 测试时，我用的最多的是场景法，占了一半以上。因为 ERP 的核心是业务链路，不是单点功能。比如出库审核，我不光测"点审核按钮能不能过"，还要测审核后库存扣没扣、凭证生没生成、金额对不对得上。场景法最适合覆盖这种多模块联动的场景。**
>
> **其次是等价类边界值——数量、金额、日期这些输入字段一定要覆盖边界。比如出库数量 = 库存量（刚好过）和 = 库存量 + 1（库存不足）这两个边界都必须测。**
>
> **然后是错误推测法，把之前线上踩过的坑加到用例里。比如"改价 + 并发出库"导致库存成本不一致这种，就是吃过亏才加进去的。**
>
> **正交实验法用得少，主要是条件组合多的时候用——比如角色 × 改价方式 × 商品类型这种，用正交表选 9 条代替 36 条。"**

---

**关键词**：用例设计方法 / 场景法 / 等价类边界值 / 错误推测法 / 正交实验法 / ERP 测试

## Q8. 你们团队用的什么开发模型？

### 问题描述

面试官问你团队用的什么开发模型（瀑布 / 敏捷 / V模型 / 迭代），考的是你**对团队协作模式的理解**，以及你在这个模式下怎么开展测试工作。

### 答案

做 ERP 项目时用的是 **Scrum 敏捷开发**，两周一个迭代。

---

### 一、我们的敏捷流程长什么样

```
两周迭代周期：

第一周                 第二周
├──周一：需求评审       ├──周一：功能测试验收
├──周二：用例设计       ├──周二：自动化回归
├──周三：用例评审       ├──周三：性能/兼容性测试
├──周四：开发提测       ├──周四：产品验收
├──周五：功能测试       ├──周五：上线 + 复盘
                        └──下午：下个迭代计划会
```

**每天**：站会（15 分钟，同步进度和阻塞）

**角色**：
- **PO（产品经理）**：提需求、排优先级
- **SM（Scrum Master）**：推动流程、解决阻塞
- **开发**：3-4 人，负责编码
- **测试（我）**：1-2 人，负责质量保障

### 二、这个模式下测试是怎么做的

#### 迭代中的测试节奏

| 时间 | 测试做什么 |
|:---:|:---|
| **迭代计划会** | 参与评估故事点，从测试角度评估复杂度 |
| **需求评审** | 理解业务逻辑，提异常场景，防漏需求 |
| **开发编码阶段** | 设计测试用例、写自动化脚本 |
| **开发提测后** | 冒烟测试 → 功能测试 → Bug跟踪 |
| **迭代末** | 自动化回归 + 上线验收 |
| **上线后** | 线上监控 + 复盘 |

#### 关键实践

**① 需求评审时测试介入**

```
❌ 以前：产品写好需求 → 开发做完 → 测试再测（经常返工）
✅ 改进后：需求评审时测试就参与
   - 从测试角度问："这个场景的异常分支是什么？"
   - 发现需求漏洞提前改，不用等做出来再返工
```

**② 开发自测 + 冒烟测试**

```
开发提测前必须：
  □ 核心链路自测通过
  □ 单元测试通过
  □ 接口文档已更新

冒烟不通过 → 打回开发，不进入详细测试
```

**③ 自动化回归是发布卡口**

```
每次迭代末：
  - 跑 90+ 条核心链路自动化用例
  - 对账脚本跑一遍（出库 vs 凭证逐笔比对）
  - 通过率 ≥ 98% 才允许上线
```

**④ 迭代复盘会**

```
每个迭代结束后的复盘（30分钟）：
  □ 这个迭代哪些做得好？
  □ 哪些问题可以改进？
  □ Bug 分布分析——哪个模块 Bug 最多？什么类型？
  □ 下个迭代的改进措施
```

### 三、如果面试官追问其他模型

| 模型 | 说明 | 和你的关系 |
|:---|:---|:---|
| **Scrum 敏捷** | 两周迭代，需求持续交付 | 你用的就是这个 |
| **Kanban** | 看板管理，不限迭代周期 | 了解即可 |
| **瀑布** | 需求→设计→开发→测试→上线，串行 | 传统项目还会用 |
| **V模型** | 测试和开发并行，每个开发阶段对应一个测试阶段 | 面试可能会问 |

**面试话术参考**：

> "我用的是 Scrum 敏捷，两周一个迭代。和瀑布比，敏捷的好处是需求变更了不用等几个月，下个迭代就能排进去。但挑战是需求变化快，测试时间被压缩，所以我们的策略是核心链路做自动化回归兜底，减少手工回归的工作量。"

### 四、面试回答模板

> **"我们团队用的是 Scrum 敏捷开发，两周一个迭代。每个迭代的节奏是：第一周需求评审 + 用例设计 + 开发编码，第二周功能测试 + 自动化回归 + 上线。**
>
> **我的测试工作完全融入这个节奏。迭代计划会参与评估故事点，需求评审时从测试角度提异常场景，开发编码阶段我同时写自动化脚本。提测后先跑冒烟，不通过就打回，通过才进入详细测试。迭代末自动化回归 90+ 条用例 + 对账检查，通过率 ≥ 98% 才上线。每个迭代结束还有复盘会，分析 Bug 分布和改进点。**
>
> **敏捷对测试的挑战是需求变化快、测试时间紧，所以我的策略是核心链路做自动化兜底，把手工回归的时间解放出来做探索性测试。"**

---

**关键词**：Scrum 敏捷 / 迭代开发 / 开发模型 / 测试节奏 / 站会 / 复盘

## Q9. 测试的工作量怎么评估？

### 问题描述

面试官问你：给你一个需求，你怎么评估测试需要多少时间？考的是你不是只会执行测试，而是**有没有评估意识**，能不能给出靠谱的时间预估。

### 答案

测试工作量评估是测试负责人必须掌握的技能。评估不准——报少了加班，报多了 PM 不答应。做 ERP 项目时，我常用的评估方法分四步走。

---

### 一、四步评估法

#### Step 1：拆分测试任务

不要拍脑袋说"这个功能要 3 天"，先拆成最小的可评估单元：

```
功能：销售改价

拆解成：
□ 用例设计       —— 设计 15 条用例（正向+异常+边界+并发）
□ 用例评审       —— 三方评审 30 分钟
□ 功能测试执行   —— 手工跑 15 条用例（Web端 + App端）
□ 自动化脚本开发 —— 把核心 5 条固化为接口自动化用例
□ 自动化调试     —— 调试通过，接入 Jenkins
□ Bug 验证       —— 留余量处理开发修的 Bug
□ 回归测试       —— 版本上线前全量回归
```

#### Step 2：逐项评估时间

用经验值估算每个子任务的时间：

```text
用例设计：
  简单功能（5-10条用例）：0.5天
  中等功能（15-30条用例）：1天
  复杂功能（30+条用例，多模块联动）：2天

用例评审：0.5天（预留排期时间）

功能测试执行：
  简单功能：0.5天
  中等功能：1天
  复杂功能：2天
  Web端 + App端 + 兼容性：再 +0.5天

自动化脚本开发：
  一个接口用例：0.5天（含调试）
  一条业务链路（3-5个接口）：1天

Bug验证：预留总工时的 20%

回归测试：0.5天（自动化回归 + 核心链路手工验证）
```

#### Step 3：加上 buffer（缓冲时间）

```text
评估出来的时间 × 1.3 = 最终报出的时间

原因：
  - 需求可能变更
  - 开发提测可能延期
  - Bug 可能比预期多
  - 环境可能不稳定
```

#### Step 4：和开发时间对比校准

```text
开发时间 vs 测试时间的经验比例：

功能类型           开发:测试
简单页面改动        1:0.5
新增功能            1:0.8-1
复杂业务链路        1:1
多模块联动的功能    1:1.2
```

如果 PM 给的开发时间是 5 天，功能涉及进销存三模块联动，测试至少要 5 × 1.2 = 6 天。如果 PM 说测试只给 2 天，就要砍范围——核心链路自动化 + 手工回归覆盖主流程，边缘场景放到下个迭代。

---

### 二、实际案例演示

以一个具体的迭代任务为例：

```
需求：出库单增加"批量审核"功能

评估过程：

1. 拆分任务
   □ 用例设计：10条（正向2 + 异常4 + 边界2 + 并发2）
   □ 用例评审
   □ 功能测试执行：Web端 + App端
   □ 自动化脚本：核心链路3条
   □ Bug验证

2. 逐项估算
   用例设计：1天
   评审：0.5天
   功能测试：1.5天（Web + App）
   自动化：1.5天（3条链路，含调试）
   Bug验证：预留 1天
   回归：0.5天

   小计：6天

3. 加 buffer
   6 × 1.3 = 7.8 ≈ 8天（报给 PM）

4. 实际执行
   过程中提测晚了1天，Bug比预期多了2个
   实际用了7.5天完成（buffer 起了作用）
```

---

### 三、快速评估公式（站会上用）

站会上 PM 问"这个功能测多久"，来不及拆解时用这个快速估算：

```text
快速估算 = 用例数 × 单用例执行时间 × 端数 × 1.3

用例数：根据需求复杂度预估
  简单：10条
  中等：20条
  复杂：30+条

单用例执行时间：0.15-0.3天（含写用例+执行+录Bug）

端数：1（单端） / 1.5（Web+App） / 2（多端+兼容性）

1.3：Buffer
```

**例**：出库批量审核，预计 15 条用例，Web+App 双端

```
15 × 0.2 × 1.5 × 1.3 = 5.85 ≈ 6天
```

---

### 四、自动化对工作量评估的影响

```
有自动化和没自动化，评估方式完全不同：

没有自动化的版本：
  每次回归都要手工跑 → 每次版本都要算回归时间
  评估 = 用例设计 + 手工执行 + 回归

有自动化的版本：
  回归时间 ≈ 0（自动化跑完看报告就行）
  评估 = 用例设计 + 手工执行 + 自动化脚本维护
```

**所以我评估时一定会问**：这次变更涉及的模块有没有自动化覆盖？

- 有自动化 → 回归时间砍掉，只算新功能测试时间
- 没自动化 → 要考虑全量回归的时间

---

### 五、面试回答模板

> **"我的工作量评估分四步。第一，把大功能拆成最小子任务——用例设计、功能执行、自动化、Bug验证、回归，每个都单独估。第二，逐项用经验值估算时间，比如中等复杂度的功能用例设计要 1 天，功能执行 Web+App 要 1.5 天。第三，加 30% 的 buffer，应对提测延迟、Bug 增多、环境问题。第四，和开发时间做对比校准——多模块联动的功能开发测试比大概 1:1.2。**
>
> **实际上线后复盘，如果实际时间和评估偏差超过 20%，我会分析原因：是需求变更了？还是漏评估了什么？下次修正。**
>
> **另外，有自动化覆盖的模块评估时可以把回归时间省掉，这也是自动化带来的核心价值之一。"**

---

**关键词**：工作量评估 / 测试估算 / 任务拆解 / Buffer / 开发测试比 / 自动化对评估的影响

## Q10. 你评估的工作量和组长评估的不一致，怎么处理？

### 问题描述

组长评估这个迭代测试要 5 天，你觉得至少要 8 天。或者反过来，组长觉得 8 天，你觉得 5 天就够了。这个时候你怎么沟通？考的是你的**职场沟通和向上管理能力**。

### 答案

这是一个非常现实的职场场景，面试官问这个问题不是考你谁对谁错，而是看你**怎么处理分歧、怎么沟通**。

我的处理方式分四步：

---

### 第一步：先对齐评估的基准（别一开始就争论数字）

很多时候分歧是因为两个人评估的基准不一样：

```text
对齐之前先确认这几点：

对齐维度           你的假设         组长的假设
─────────────────────────────────────────────
是否含 buffer？     含了30%          没含
测试范围？          Web端+App端      Web端
自动化覆盖？        手工回归         已自动化不用测
兼容性？            包含3个浏览器    只测Chrome
Bug预留？           预留了20%        没预留
```

**实际案例**：

> 有次我评估出库批量审核要 8 天，组长说 5 天就够了。对齐后发现组长只算了 Web 端，没算 App 端。加上 App 端测试 1.5 天，再加上兼容性 0.5 天，他改口到 7 天，基本和我一致了。**分歧是因为对范围的认知不一样，不是真的对不上。**

### 第二步：找到具体的差异点在哪里

如果基准对齐了还是不一致，拆开看差异在哪个子任务：

```text
我的评估：                   组长的评估：
  用例设计：1天               用例设计：0.5天
  功能测试：2天               功能测试：1.5天
  自动化脚本：1.5天           自动化脚本：1天
  Bug验证：1天                Bug验证：0.5天
  回归：0.5天                 回归：0.5天
  环境搭建：0.5天             （没算）
  Buffer：1.5天               （没算）
──────────────────────────────
合计：8天                    合计：5天

差异点：
  1. 组长没算环境搭建（他说环境已经好了）
  2. 组长没算 buffer（他说按理想情况排）
  3. 组长觉得自动化脚本 1 天就够了（我觉得新接口要调 1.5 天）
```

### 第三步：用数据和经验说话，不要凭感觉

```text
不要说："我觉得这个要 3 天"
要说："上次类似的改价功能，有 12 条用例，Web+App 跑了 2.5 天。
       这次出库批量审核场景更复杂，涉及三模块联动，预计 15 条用例，
       按上次的节奏至少 3 天。"
```

**用历史数据比用感觉更有说服力**。

### 第四步：如果还有分歧，提出折中方案

对不齐的时候不要硬刚，给出可选择的方案：

```text
方案A：按 8 天排（覆盖全部 + 自动化 + buffer）
  优点：质量有保障
  缺点：工期长

方案B：按 5 天排，但砍范围
  砍掉：App端测试（留到下个迭代）
  砍掉：兼容性测试（只测 Chrome）
  砍掉：自动化脚本（下个迭代再补）
  风险：App端如果有 Bug 上线后才能发现

方案C：按 6 天排（折中）
  保：Web端功能测试 + 核心链路自动化
  砍：App端放到下个迭代
  不加 buffer，但如果有延期风险要提前沟通
```

**让组长做选择题，而不是问答题**。

---

### 实战话术模板

> **场景 1：组长估少了，你觉得不够**

```
"组长，我拆了一下这个功能的任务，发现我的评估和你差 3 天。
我列了个对比表——我的 8 天含了 App 端 1.5 天和环境搭建 0.5 天，
你那边是不是默认 App 端不用测？

如果可以砍掉 App 端，我这边 6.5 天就能搞定。
如果不砍的话，我建议加 1 天 buffer，7 天比较稳。"
```

> **场景 2：组长估多了，你觉得不需要那么久**

```
"组长，我看了下这个功能，之前类似的改价功能我做过，
Web端 15 条用例测了 2.5 天。这次比上次还少了兼容性，
我评估 5 天就够了，不需要 8 天。
如果按 5 天排，我可以提前开始下个迭代的用例设计。"
```

### 面试回答模板

> **"如果我和组长的评估不一致，我不会直接说'你不对'，而是先对齐评估基准。很多时候分歧是因为两个人对范围、buffer、自动化的假设不一样。**
>
> **比如有次我评估 8 天、组长说 5 天，对齐后发现他只算了 Web 端没算 App 端，加上之后就基本一致了。**
>
> **如果基准对齐了还是不一样，我就拆开看差异在哪个子任务——是自动化估多了？还是 bug 验证估少了？用上一次的历史数据说话，比凭感觉更有说服力。**
>
> **最后如果还是有分歧，我会给几个方案让他选："
按多的排质量有保障，按少的排就砍范围，或者中间取个值。"**

**关键词**：工作量分歧 / 向上沟通 / 评估对齐 / 折中方案 / 职场沟通

## Q11. 你提了一个 Bug，开发不认为是 Bug，怎么沟通？

### 问题描述

测试提了一个 Bug，开发说"这不是 Bug，就是这么设计的"或者"这个不影响功能，不用改"。你怎么说服开发？考的是你的**沟通能力和专业判断力**。

### 答案

这是测试工作中最常遇到的场景，处理方式分几步走。

---

### 第一步：先确认是不是自己判断错了

不要一上来就硬怼，先冷静判断：

```
自检清单：
□ 这个 Bug 我有明确的证据吗？（截图/日志/接口返回）
□ 我理解需求文档吗？有没有可能是需求就是这么设计的？
□ 是不是我的测试环境和开发环境不一致？
□ 是不是测试数据和生产数据不一样导致的？
```

**自检通过再找开发，自检没通过先自己搞清楚。**

### 第二步：带着证据去沟通，不带情绪

不是一个"你觉得、我觉得"，而是用数据和证据说话。

```text
❌ 不要这样说：
"这个出库审核报 500 了，你快修一下。"
"这个就是 Bug，你不认也得认。"

✅ 要这样说：
"这个出库审核接口，我传了正确的参数，返回了 500。
截图在这里，日志在这里——grep -C 10 搜到了报错信息。
你看是不是你们那边有什么异常？"
```

**证据清单**（按说服力排序）：

| 证据类型 | 说服力 | 说明 |
|:---:|:---:|:---:|
| 抓包截图 | ⭐⭐⭐⭐⭐ | 接口返回 500 或数据不对，直接截图 |
| 服务器日志 | ⭐⭐⭐⭐⭐ | grep -C 10 搜出报错上下文 |
| 需求文档对照 | ⭐⭐⭐⭐ | "需求上写的是 A，实际跑出来是 B" |
| 数据库查询 | ⭐⭐⭐⭐ | 数据状态和预期不一致 |
| 操作录屏 | ⭐⭐⭐ | 肉眼可见的异常 |
| 口头描述 | ⭐ | 你说"我点了没反应"——没用 |

### 第三步：如果开发说"就是这么设计的"

这是最常见的拒绝理由。你要判断他是**真·就是这么设计的**还是**不想改**。

```text
问三个问题来判断：

Q1："需求文档/原型上有写吗？"
   → 如果需求上明确这么写 → 不是 Bug，是需求如此
   → 如果需求上没写或写了相反 → 继续问

Q2："那用户能接受这个表现吗？"
   → 站在用户角度思考，不是站开发角度
   "用户出库审核看到 500 报错，他应该怎么办？"

Q3："如果不出这个问题，会有什么后果？"
   → 上升到业务影响
   "如果出库审核失败用户没发现，月底对账不平怎么办？"
```

### 第四步：按严重程度决定处理力度

| 程度 | 怎么做 |
|:---|:---|
| **P0/P1** — 阻塞主流程、影响核心业务 | 坚持必须修，升级到组长/PM |
| **P2** — 有替代方案、影响体验 | 可以协商排到下个迭代 |
| **P3** — 界面文案、体验问题 | 可以接受延期或不修 |

### 第五步：如果还是不认，升级处理

如果你确定是 Bug，开发就是不认，按这个顺序升级：

```text
第1次：和开发本人沟通 → 带证据、讲道理
第2次：和开发 Leader 沟通 → "这个功能 P2 级别 Bug，和开发 XX 沟通过，他暂时排不进来"
第3次：产品经理确认 → "产品你看这个表现和需求一致吗？"
第4次：测试组长介入 → 组长和开发组长对话
```

**注意**：不要跨级告状，按顺序来。

---

### 实战案例

> **场景**：测试出库审核功能，发现审核通过后，凭证金额少了一分钱。
>
> **开发**："这一分钱是精度问题，不影响使用，不用改。"
>
> **我的处理**：
>
> 第一步，先自己确认——查了数据库，确实出库单金额是 100.00，凭证是 99.99。有截图有 SQL 记录。
>
> 第二步，找开发沟通——"这个金额不对，出库 100 凭证 99.99，月底对账会差一分。虽然一分钱不多，但财务那边对账是逐笔核对的，每一笔都要对得上。"
>
> 第三步，开发还是不认——"这是浮点数精度问题，改起来成本高。"
>
> 第四步，升级——我找了产品确认，产品说"财务要求必须精确到分，不能有误差"。然后产品和开发沟通后，开发改了精度处理逻辑。
>
> 最后改了，其实就是 `round(amount, 2)` 的问题，5 分钟修好。

---

### 面试回答模板

> **"遇到开发不认 Bug 的情况，我分四步处理。**
>
> **第一，先自检——确认不是自己环境问题、不是测试数据和生产数据差异、不是需求本身就是这个设计。**
>
> **第二，带证据沟通——截图、日志、接口返回、数据库查询结果，一样样摆出来。不说'你这个有 Bug'，说'这个接口返回 500，你看是不是异常'。**
>
> **第三，如果开发说'就是这么设计的'，我反问三个问题：需求文档上有写吗？用户能接受吗？不出会有什么后果？上升到业务影响来沟通。**
>
> **第四，如果还是不认，P0/P1 的 Bug 我会坚持升级——按顺序找开发组长、产品、测试组长介入。P2/P3 的可以协商排期。**
>
> **核心原则：对事不对人，用证据说话，上升到业务价值而不是纠结技术细节。"**

---

**关键词**：Bug 沟通 / 开发不认 Bug / 职场沟通 / 升级处理 / 证据驱动

## Q12. 怎么判断测试用例的优先级？

### 问题描述

面试官问你：比如你有 100 条测试用例，时间只够跑 30 条，你先跑哪些？怎么判断哪些用例优先级高、哪些可以缓一缓？

### 答案

用例优先级不是拍脑袋定的，我一般从 **四个维度** 来判断：

> **业务影响 > 用户使用频率 > 风险程度 > 覆盖范围**

---

### 一、四维评估法

#### 维度 1：业务影响（权重最高）

这个功能出问题会造成什么后果？

```text
P0 — 核心业务，出问题直接导致业务无法进行
  例：登录、创建订单、出库审核、支付
  → 优先级最高，必须测

P1 — 重要功能，出问题有替代方案但影响大
  例：改价、库存查询、报表导出
  → 优先级高，尽量测

P2 — 辅助功能，出问题不影响主流程
  例：历史订单查看、个人设置
  → 有时间就测

P3 — 体验优化，出问题不直接影响功能
  例：按钮位置、文案错误、颜色不对
  → 最后测
```

#### 维度 2：用户使用频率

同样重要程度下，用户用得多的优先测：

```text
高频（每天用）：出库审核、入库、库存查询        → P0
中频（每周用）：报表导出、调拨、批量改价         → P1
低频（偶尔用）：历史数据归档、系统配置           → P2
几乎不用：管理员设置、系统日志查看               → P3
```

#### 维度 3：风险程度

这个功能改了什么东西？改动越大风险越高：

```text
高风险（优先级最高）：
  - 新增了功能/模块
  - 修改了核心业务逻辑
  - 修改了数据库表结构
  - 涉及多个模块联动

中风险：
  - UI 改版（逻辑没变）
  - 新增查询条件

低风险：
  - 文案修改
  - 字段位置调整
```

#### 维度 4：覆盖范围

这条用例覆盖了多少场景？

```text
高覆盖用例（优先级高）：一条用例覆盖了多个业务场景
  "创建订单 → 审核 → 出库 → 凭证生成" → 一条用例跑通整条链路

低覆盖用例（优先级低）：只覆盖了一个边角场景
  "修改按钮颜色" → 只有这一个检查点
```

---

### 二、ERP 业务中的实际分级

以你的 ERP 进销存为例：

```text
P0 — 冒烟测试（每次版本必须跑通）
  □ 登录成功
  □ 创建销售单 → 审核 → 出库 → 凭证生成（正向全链路）
  □ 采购入库 → 库存更新
  □ 核心接口返回 200（接口自动化冒烟）

P1 — 核心功能测试（尽量全部覆盖）
  □ 销售改价 → 库存成本同步
  □ 出库审核 → 凭证生成
  □ 单据作废 → 库存回滚 → 凭证冲红
  □ 调拨单 → 两仓库存总和不变
  □ App 端核心操作

P2 — 辅助功能测试（有时间就测）
  □ 列表分页查询
  □ 条件筛选
  □ 报表导出
  □ 历史数据查看

P3 — 体验类测试（最后测）
  □ 按钮样式
  □ 文案错误
  □ 提示信息是否友好
```

### 三、时间不够时怎么砍用例

当你只有 30% 的时间时：

```text
保留：P0 全部（核心链路 + 冒烟）
砍掉：P3 全部（体验类）
协商：P2 部分（和 PM 确认哪些可以延迟到下次）

优先级铁律：
  永远优先覆盖核心业务链路
  一条"正向全链路"用例 > 十条"边角输入框验证"用例
```

### 四、面试回答模板

> **"我判断用例优先级看四个维度。第一是业务影响——这个功能出问题会造成什么后果？出库审核报 500 是 P0，按钮颜色不对是 P3。第二是用户使用频率——出库审核每天用，历史归档几个月用一次，前者优先级高。第三是风险程度——改了核心逻辑的还是只改了文案的。第四是覆盖范围——一条覆盖整条链路的用例比十条边角用例优先级高。**
>
> **时间不够时我的原则是：核心链路全量覆盖 + 边缘场景排优先级。P0 一条不能少，P3 全砍，P2 和 PM 协商哪些可以延迟。"**

---

**关键词**：用例优先级 / 业务影响 / 冒烟测试 / 核心链路 / 测试范围

## Q13. 开发没通知就打包了下个版本的功能，QA 环境又没法回滚，怎么处理？

### 问题描述

开发在没告知测试的情况下，把下个版本的功能也打包上去了，QA 环境又没办法回滚到应该测的那个版本。这个时候你怎么处理？

### 答案

这个场景在测试工作中并不少见，核心问题是**版本管理不规范 + 环境回滚能力缺失**。处理方式分当前怎么办和以后怎么预防两步走。

---

### 一、当前怎么办（应急处理）

#### 方案 A：如果下个版本的功能冲突不大

直接在这个环境上测，但要做**版本偏差记录**：

```text
例：本来要测 v2.3 的出库审核，但环境上已经是 v2.4（含了下个版本的改价功能）

处理方法：
  1. 和开发确认 v2.4 相比 v2.3 多了哪些东西
  2. 如果新增功能不影响本次测试范围 → 忽略，正常测
  3. 但必须在测试报告中写明：
     "本次测试基于 v2.4 版本，较计划版本 v2.3 新增了 XXX 功能。
     如后续回滚到 v2.3，建议对涉及模块补测。"
```

#### 方案 B：如果下个版本的功能会干扰本次测试

比如下个版本改了数据库表结构，导致当前版本的用例跑不通：

```text
1. 第一时间找开发 Leader 反馈
   "QA 环境被 v2.4 覆盖了，v2.3 没法回滚，
   我们的测试用例是基于 v2.3 数据库结构的，现在跑不通了。
   需要你们帮忙解决环境问题。"

2. 同时找测试组长同步风险
   "环境被覆盖，测试可能延期，建议走风险登记。"

3. 如果开发能解决：
   让他们重新部署 v2.3 或修复兼容性问题

4. 如果解决不了（环境真回滚不了）：
   降级方案：在开发本地环境测（让开发搭一套 v2.3）
   或者：用测试的本地 Docker 环境
   或者：这次只做代码 Review + 接口冒烟，全量测试推到下个版本
```

#### 方案 C：临时搭一套可测环境

```bash
# 如果项目有 Docker 化部署，可以自己在开发机上搭
docker-compose -f docker-compose.v2.3.yml up -d

# 或者让开发从代码仓库 checkout v2.3 标签，本地部署
git checkout v2.3
mvn clean package -DskipTests
# 本地启动，测试连本地环境测
```

---

### 二、以后怎么预防

这次的问题虽然解决了，但根因是**版本管理不规范**。需要推动改进：

#### 改进 1：版本命名规范 + 部署流程

```text
规范：
  - 每个版本必须有明确的 tag（v2.3.0 / v2.4.0）
  - 部署 QA 环境前，开发必须在群里通知测试
    "@测试 v2.3.0 已部署到 QA 环境，可以开始测了"
  - 未经测试确认，不允许往 QA 环境部署新版本
```

#### 改进 2：QA 环境支持回滚

```text
推动 DevOps 或开发配合：
  - 每次部署前自动备份当前版本（docker save / 备份 jar 包）
  - 支持一键回滚到上一个版本
  - 保留至少最近 3 个版本的部署包

如果公司没条件（小团队）：
  - 至少保留上一次的部署包，手动回滚
  - 或者用 Docker 标签管理版本
```

## Q14. 测试完上线后又发现了问题，怎么做？

### 问题描述

测试环境跑了好几轮回归，全绿，上线后业务反馈还是出了问题。这个时候你怎么处理？考的是你的**线上事故处理能力和复盘改进能力**。

### 答案

上线后出问题是测试最不愿意面对但一定会遇到的情况。处理分四步：止血 → 定位 → 修复 → 复盘。

---

### 第一步：止血（先控制影响范围，不追责）

出了线上问题，第一件事不是找谁的责任，而是先把损失降到最低。

```text
止血动作：

1. 判断严重程度
   P0：核心业务不可用 → 立即回滚版本
   P1：功能异常但有替代方案 → 评估是否回滚
   P2：非核心功能异常 → 记录，下个版本修

2. 回滚（最安全的方式）
   如果影响大，直接回滚到上一个稳定版本
   等修好了再重新上线

3. 降级（如果不能回滚）
   关掉出问题的功能开关（feature toggle）
   或者切换到备用逻辑

4. 通知相关人员
   通知测试组长、开发、PM
   "线上发现 XXX 问题，已回滚到 v2.2，正在排查根因"
```

---

### 第二步：定位（找出根因）

止血之后，开始排查为什么测试没发现。

#### 常见的「测试没发现」根因

```text
原因分类                    具体说明
─────────────────────────────────────────────────────
测试环境 ≠ 生产环境         数据量不同、配置不同、硬件不同
测试数据覆盖不全           生产上有测试环境没有的数据状态
并发场景没覆盖             QA 单人操作，线上多用户并发
时间维度问题               凌晨日切、定时任务、缓存过期
配置差异                   QA 和生产的配置参数不同
灰度/AB测试逻辑            QA 只有一个版本，线上有多个版本在跑
第三方依赖                 QA 的 mock 返回和真实第三方不一样
```

#### 排查方法

```text
1. 查日志
   grep -C 10 "错误信息" /var/log/app/app.log
   看线上报错和测试环境的日志有什么不同

2. 对比数据
   用生产的数据在测试环境复现一次
   看能不能重现

3. 确认环境差异
   检查 QA 和生产的配置、数据库结构、中间件版本
```

---

### 第三步：修复（临时 + 长期）

```text
临时修复：
  - 修 Bug 后，热修复版本走紧急上线流程
  - 验证：在测试环境用生产数据复现并验证通过

长期修复：
  - 把这个场景补到测试用例里
  - 如果是环境差异导致的 → 推动测试环境对齐生产
  - 如果是并发没覆盖 → 补并发测试
  - 如果是配置差异 → 加配置检查步骤
```

---

### 第四步：复盘（防止再犯）

复盘不是为了追责，而是为了**下次不再发生同样的问题**。

#### 复盘模板

```text
┌──────────────────────────────────────────────┐
│          线上事故复盘报告                       │
├──────────────────────────────────────────────┤
│ 问题描述：出库审核后凭证未生成                   │
│ 发现时间：2026-06-21 10:00                    │
│ 影响范围：5 笔出库单未生成凭证                   │
│ 根因：QA 环境配置的凭证生成超时是 30s            │
│       生产配置的是 5s，第三方接口偶发超过 5s     │
│                                              │
│ 测试为什么没发现：                              │
│   - QA 环境超时配置和生产不一致                  │
│   - 没有覆盖"超时场景"的用例                     │
│                                              │
│ 改进措施：                                     │
│   1. QA 环境配置对齐生产（已完成）               │
│   2. 增加"凭证生成超时"异常场景用例（本期补）     │
│   3. 上线前增加配置比对检查（推动 DevOps）       │
│                                              │
│ 责任人：不追责，重点关注流程改进                 │
└──────────────────────────────────────────────┘
```

#### 复盘会上的态度

```text
✅ 正确态度：
  "这次是我们测试没覆盖到超时场景，我已经补了用例，
  并且推动了 QA 环境配置对齐生产。下次不会再漏了。"

❌ 错误态度：
  "这是开发配置不对，关我什么事。"
  "测试环境就是这样，没办法和生产一模一样。"
```

---

### 实战案例

> **背景**：上线后发现出库审核成功后，凭证生成偶尔失败。QA 测试时是全绿的。
>
> **排查**：
> 1. 先止血——通知回滚到上一个版本
> 2. 查日志——发现凭证生成接口超时了
> 3. 对比环境——QA 配置的超时时间是 30s，生产是 5s
> 4. 根因——第三方支付接口偶发超过 5s，QA 环境因为超时时间长从来没触发过
>
> **改进**：
> 1. QA 环境超时配置改成和生产一致（3s）
> 2. 增加"凭证生成超时"的异常场景用例
> 3. 上线前置步骤增加"配置比对检查"

---

### 面试回答模板

> **"上线后发现问题，我分四步处理。第一是止血——P0 问题立即回滚，P1 评估是否回滚，先控制影响范围。第二是定位——查日志、对比测试环境和生产环境的差异，找到测试为什么没发现。常见原因有环境配置不同、数据覆盖不全、并发场景没覆盖。第三是修复——紧急修 Bug 走热修复流程，同时把这个场景补到用例里。第四是复盘——写复盘报告，不追责，重点找流程漏洞，推动改进。**
>
> **核心原则：线上出问题不可怕，可怕的是同一个问题出两次。确保每次复盘都有改进措施落地。"**

---

## Q16. 回归用例怎么挑选？

### 问题描述

面试官问你：回归测试不是全量跑，你从 160 条用例里怎么选出要回归的那几十条？

### 答案

回归用例不是拍脑袋选的，按 **四个原则** 来挑：

> **核心链路必跑 > 改了哪些测哪些 > 高频场景必跑 > 曾经出过问题的加进回归**

---

### 一、核心链路（必跑，最高优先级）

**核心链路 = 业务从开始到结束的一条完整链路。** 不管改了什么，这条链路必须跑通。

以 ERP 进销存为例：

```text
#1 采购核心链路：
  创建采购单 → 审核 → 入库 → 库存增加 → 生成应付凭证

#2 销售核心链路：
  创建销售单 → 审核 → 扣库存 → 生成出库单 → 出库审核 → 生成凭证

#3 调拨核心链路：
  创建调拨单 → 审核 → A仓出库 → B仓入库 → 两仓库存总和不变

#4 作废回滚链路：
  创建销售单 → 审核 → 作废 → 库存回滚 → 凭证冲红

≈ 4 条核心链路，每条展开 2-3 条用例，大约 10 条
```

**特点**：全自动化，每次部署就跑

### 二、改了哪些测哪些（最高效）

本次版本改了什么东西，就重点回归相关模块：

```text
本次改了什么              回归范围
─────────────────────────────────────────────
改了销售改价接口           销售模块全部 40 条用例
改了出库审核逻辑           出库审核相关 15 条 + 凭证联动 10 条
改了数据库表结构           所有涉及该表的用例
只改了前端 UI              前端的相关页面 5 条
只修了一个 Bug             只回归该 Bug 的用例
```

### 三、高频场景（每天都用的功能）

```text
用户日常高频使用的功能，即使没改也要回归：

  销售：创建订单、出库审核
  库存：库存查询
  财务：凭证生成、金额校验
  登录：所有角色的登录
```

### 四、曾经出过问题（踩过坑）

每次线上出过问题的场景，必须加进回归用例库里：

```text
历史上线出过的问题：
  1. 出库审核后凭证未生成 → 回归必跑
  2. 作废后库存未回滚 → 回归必跑
  3. 改价后成本未同步 → 回归必跑
  4. 并发操作导致金额差异 → 回归必跑
```

### 五、实操：160 条用例选回归

以 160 条用例为例，回归的挑选过程：

```text
第一轮挑（冒烟回归 ~10 条）：
  选 4 条核心链路，必跑

第二轮挑（核心回归 ~50 条）：
  4 条核心链路 × 扩展分支（异常场景、边界） → 20 条
  本次变更涉及的功能 → 15 条
  高频日常功能 → 10 条
  历史 Bug 场景 → 5 条

第三轮挑（全量回归 ~160 条）：
  全量，大版本跑

排除：
  纯 UI 样式检查 → 不回归
  低频功能（半年用一次） → 不回归
  探索性测试 → 不回归
```

### 六、面试回答模板

> **"回归用例我按四个原则挑。第一是核心链路——不管改了什么，4 条端到端链路必须跑通，大约 10 条用例。第二是改了什么测什么——本次改了出库审核，就重点回归出库模块 15 条和凭证联动 10 条。第三是高频场景——每日常用的功能即使没改也要回归。第四是踩过的坑——线上出过问题的场景必须加进回归。**

### 答案

库存模块是 ERP 最复杂的模块之一，我把它拆成 **8 个维度** 来覆盖：

---

### 一、库存变更的正确性

```text
入库增加 → 采购入库后库存量 +N
出库减少 → 销售出库后库存量 -N
调拨转移 → A仓库存 -N，B仓库存 +N
盘点调整 → 盘盈/盘亏后库存修正
作废回滚 → 作废单据后库存恢复原值
```

**核心断言**：`操作后的库存量 == 操作前的库存量 ± 变更量`

---

### 二、多仓库隔离

```text
□ 查询 WH001 的库存时，不出现 WH002 的数据
□ 调拨时 A仓扣、B仓加，两边各自独立
□ 创建单据时仓库选择默认为用户所属仓库
□ 不同仓同一 SKU 互相不干扰
```

---

### 三、库存状态流转

```text
可用库存 → 锁定库存（订单审核后扣减）
         → 占用库存（调拨中）
         → 冻结库存（盘点中、质检中）

重点测：
  □ 状态之间不能跳跃——已出库的不能锁定，锁定的不能再锁定
  □ 锁定库存也占用总量——可售的只有"可用库存"
```

---

### 四、数据一致性（和上下游对账）

这是库存测试**最重要的点**，也最容易出偶发 Bug：

```text
□ 出库单数量 = 库存扣减量
□ 入库单数量 = 库存增加量
□ 调拨守恒：A仓扣减量 = B仓增加量 → 两仓总和不变
□ 作废回滚：回滚后库存 = 操作前库存
□ 改价后库存成本同步更新
```

---

### 五、并发场景

```text
□ 多人同时出库同一 SKU → 会不会超卖？
□ 一人出库、另一人盘点 → 数据不互相覆盖
□ 一人调拨+另一人出库 → A仓已扣但B仓未到就被出库？
□ 同时操作多仓同一 SKU → 锁机制是否生效？
```

---

### 六、边界值

```text
□ 库存量 = 0        → 出库审核不通过
□ 库存量 = 1        → 出库 1 刚好
□ 库存量 = 999999   → 大数据量下页面是否卡顿
□ SKU 数量：1个 / 100个 / 10000个
□ 出库数量 > 库存量  → 审核不通过
```

---

### 七、库存预警

```text
□ 安全库存设置 → 低于阈值时系统提示
□ 库存不足时下发拦截（不让创建超过库存的出库单）
□ 断货预警 → 通知采购补货
```

---

### 八、性能

```text
□ 10 万 SKU 的库存查询页面加载速度
□ 批量出入库时的接口响应时间
□ 库存报表大数据量导出
```

---

### 九、面试回答模板

> **"库存模块我重点关注八个维度。最核心的是数据一致性——每条变更后库存量的加减是否正确，出库扣 10 个库存必须少 10 个，调拨后两仓总和不变。这是线上最容易出偶发不同步的地方，所以我全部固化为接口自动化用例加对账断言。**
>
> **其次是并发——多人同时出库同一 SKU 会不会超卖，调拨和出库同时操作会不会数据覆盖。然后是状态流转——锁定了不能再锁、已出库的不能回退。这些核心场景每次回归必跑。"**

---

**关键词**：库存模块 / 功能点 / 数据一致性 / 并发 / 库存状态 / 多仓库

## Q18. 库存准确性（不丢数、不超卖）你们怎么保障的？

### 问题描述

面试官追问：库存最怕两个问题——数据不对（丢了/多了）和超卖（卖出去了库存没扣）。你是负责进销存测试的，你们用什么体系来保障库存准确性？

### 答案

库存准确性不是靠"测试阶段多跑几条用例"就能解决的，我们用了 **四层保障体系**：

> **用例覆盖 → 自动化对账 → 并发校验 → 线上巡检**

---

### 一、用例覆盖：核心链路必测

```text
每条版本上线前必跑：

□ 正向：采购入库 → 库存 +100 → 验证查询显示 100
□ 正向：销售出库 → 库存 -50 → 验证查询显示 50
□ 回滚：销售单作废 → 库存 +50 → 验证查询恢复 100
□ 调拨：A→B 50个 → 验证 A-50 B+50，总和不变
□ 盘点：盘盈/盘亏 → 验证库存修正到盘点值
```

---

### 二、自动化对账（最关键的一层）

**单独一条接口返回 200 不能保证库存正确**，必须跨接口做数据对账：

```python
def test_outbound_deduct_stock(api_client):
    """出库审核后，库存是否真正扣减了"""

    # Step 1：操作前查库存
    stock_before = api_client.get("/wms/stock/query", params={"sku": "SKU001"}).json()["quantity"]

    # Step 2：创建并审核出库单
    order = api_client.post("/sale/order/create", json={"sku": "SKU001", "quantity": 10}).json()
    api_client.post(f"/sale/order/{order['id']}/audit")
    api_client.post(f"/wms/outbound/{order['id']}/audit")

    # Step 3：操作后查库存
    stock_after = api_client.get("/wms/stock/query", params={"sku": "SKU001"}).json()["quantity"]

    # Step 4：对账断言
    assert stock_after == stock_before - 10, \
        f"❌ 库存扣减错误！操作前={stock_before} 操作后={stock_after} 应扣=10"
```

**这条对账用例接入 Jenkins 每天跑，发现不对直接告警。**

---

### 三、并发场景专门测：防止超卖

超卖的根因是并发——两个人同时买最后一件，库存都显示为 1，两个都卖了。

```python
import threading

def test_no_oversell(api_client):
    """并发出库：库存 10，50 人每人买 1 个，只有前 10 个成功"""

    # 初始化库存
    api_client.post("/testdata/init_stock", json={"sku": "SKU_CONCURRENT", "quantity": 10})

    results = []
    lock = threading.Lock()

    def create_and_audit(user_id):
        resp = api_client.post("/sale/order/create", json={"sku": "SKU_CONCURRENT", "quantity": 1})
        order_id = resp.json()["id"]
        audit = api_client.post(f"/sale/order/{order_id}/audit")
        with lock:
            results.append(audit.status_code)

    # 50 个线程并发出库
    threads = [threading.Thread(target=create_and_audit, args=(i,)) for i in range(50)]
    for t in threads: t.start()
    for t in threads: t.join()

    # 验证只有 10 个成功（库存只有 10）
    success = sum(1 for r in results if r == 200)
    fail = sum(1 for r in results if r != 200)
    assert success == 10, f"❌ 超卖了！成功={success} 应=10"
    assert fail == 40, f"❌ 失败数不对: 失败={fail} 应=40"

    # 验证库存为 0（全卖完了）
    final = api_client.get("/wms/stock/query", params={"sku": "SKU_CONCURRENT"}).json()
    assert final["quantity"] == 0, f"❌ 库���没扣完: {final['quantity']}"
```

---

### 四、线上巡检：业务反馈之前发现

自动化回归只覆盖了已知场景，线上还有偶发问题。所以做了定时对账巡检：

```sql
-- 每小时跑一次，检查异常库存
SELECT sku_code, warehouse_id, quantity
FROM wms_stock
WHERE quantity < 0;  -- 库存为负 = 超卖了

SELECT o.id, o.sku_code, o.quantity
FROM sale_order o
WHERE o.status = 'DONE'
  AND o.created_at > NOW() - INTERVAL 2 HOUR
  AND NOT EXISTS (
    SELECT 1 FROM wms_stock_log l
    WHERE l.biz_type = 'outbound' AND l.biz_id = o.id
  );  -- 已完成的订单没有扣库存记录
```

**巡检发现异常 → 钉钉推送告警 → 人工介入 → 同步写对账修复脚本**

```python
# 自动修复脚本
def auto_fix_negative_stock():
    """发现库存为负时，通知并记录，不要自动改（财务数据不能自动动）"""
    negatives = db.query("SELECT * FROM wms_stock WHERE quantity < 0")
    for row in negatives:
        notify(f"⚠️ 库存异常: {row.sku_code} 库存={row.quantity}")
        log_anomaly(row)  # 不自动修，等人工确认
```

---

### 五、四层体系总结

```
第一层：用例覆盖       → 每次版本必跑核心链路
第二层：自动化对账       → 跨接口验证库存加减是否正确
第三层：并发校验       → 50 线程模拟，精确验证不会超卖
第四层：线上巡检       → 每小时扫一次负数库存，业务发现前就知道了
```

### 六、面试回答模板

> **"库存准确性的保障我们做了四层。第一是每次版本上线跑核心链路——入库、出库、调拨、作废，每步验证库存量的变化。**
>
> **第二是自动化对账——不能只断言接口 200，必须跨接口验证。出库接口返回成功不等于库存真扣了，我每一步都查出库单的量、查库存的量、做减法比对。这条对账用例接入了 Jenkins 每天跑，发现问题直接告警。**
>
> **第三是并发校验——写了一个并发脚本，50 线程同时出库一个只有 10 个库存的商品，精确验证只有前 10 个成功，多一个就是超卖。**
>
> **第四是线上巡检——每小时扫一次负数库存和已成交未扣库存的单据，在业务发现之前就已经在查了。"**

---

**关键词**：库存准确性 / 超卖 / 并发测试 / 对账 / 线上巡检 / 负数库存

## Q19. SQL 注入和 XSS 安全测试怎么做？

### 问题描述

面试官问你做过安全测试吗？你说没做过就没分，但你可以说做过基础的安全扫描。

### 答案

做 ERP 项目时主要关注两类：**SQL 注入**和**XSS 跨站脚本**。虽然公司有专门的安全团队，但测试阶段要做基础的安全校验。

### 一、SQL 注入测试

**原理**：把恶意 SQL 拼到输入框里，试图操控数据库。

```python
# 自动化用例——尝试 SQL 注入
def test_sql_injection(api_client):
    """SQL 注入应被拦截，返回参数错误而非执行恶意 SQL"""
    payloads = [
        "admin' OR '1'='1",           # 万能密码
        "admin' --",                  # 注释掉后面条件
        "admin'; DROP TABLE users;--",# 删表（危险）
        "' OR 1=1 --",
    ]
    for payload in payloads:
        resp = api_client.post("/auth/login", json={
            "username": payload,
            "password": "123456"
        })
        assert resp.status_code != 200, f"❌ SQL注入未拦截: {payload}"
        assert resp.status_code in [400, 401, 403], f"❌ 期望拦截 {payload}"
```

**防御**：
- 参数化查询（不用字符串拼接 SQL）
- 输入校验 + 转义
- 最小权限原则

### 二、XSS 跨站脚本测试

```python
def test_xss_prevention(api_client):
    """XSS 脚本应被转义，不能直接执行"""
    payloads = [
        "<script>alert('xss')</script>",
        "<img src=x onerror=alert(1)>",
        "<body onload=alert('xss')>",
    ]
    for payload in payloads:
        resp = api_client.post("/sale/order/create", json={
            "remark": payload  # 输入恶意脚本到备注字段
        })
        # 查询返回的数据，脚本应被转义为 &lt;script&gt;
        order = api_client.get(f"/sale/order/{resp.json()['id']}").json()
        assert "<script>" not in order["remark"], "❌ XSS 未转义"
```

**防御**：
- 前端渲染时转义 HTML 特殊字符
- 设置 Content-Security-Policy 头
- 输入白名单校验

### 三、其他基础安全校验

```text
□ 越权测试：普通用户调用管理员接口 → 应返回 403
□ 接口频率限制：短时间内大量请求 → 应触发限流
□ 敏感信息泄露：接口返回中是否有密码明文？
□ 文件上传：上传可执行文件 → 应拦截
□ Token 过期：过期 token 能否继续用？
```

>
> **自动化用例里写了一个安全扫描 fixture——登录、订单创建、备注字段，都跑一遍 SQL 注入和 XSS payload，有漏的直接标记为 Bug。"**

## Q21. 你负责的采购、销售、库存三模块，各自的测试难点是什么？

### 问题描述

面试官问你：你做 ERP 进销存三模块，每个模块最难测的是什么？考的是你有没有深入理解业务，而不是停留在"点按钮看对不对"。

### 答案

三个模块的难点各不相同，但核心都绕不开 **数据一致性**：

---

### 一、采购模块难点：入库状态流转 + 应付对账

**难点 1：采购入库的状态机和实物匹配**

```text
采购单状态：待审核 → 已审核 → 部分入库 → 全部入库 → 已完成
              ↘ 已作废

测试难点：
  1. "部分入库"到"全部入库"的状态切换临界点
     → 采购 100 个，第一次入 99 个→ 部分入库
     → 第二次入 1 个 → 是否自动变更为"全部入库"？
  2. 入库数量 > 采购数量 → 必须拦截
  3. 入库后采购单审核还能不能改数量？→ 不能
```

**难点 2：应付账款对账**

```text
采购入库后，财务模块是否生成应付凭证？
凭证金额 = 入库数量 × 入库单价？
如果分批入库，凭证是每批单独生成还是汇总？
退货后，应付凭证是否冲红？

测试方法：
  对账断言：采购入库金额 vs 应付凭证金额逐笔比对
  abs(入库金额 - 凭证金额) < 0.01
```

---

### 二、销售模块难点：改价联动 + 并发扣库存

**难点 1：改价后的成本同步（最头疼）**

```text
场景：商品 SKU001 原价 100 元，销售中调价到 150 元

测试难点：
  □ 改价后，历史未完成的订单金额是否受影响？
  □ 改价后，已出库的订单金额会不会跟着变？（应该不会）
  □ 改价后，库存成本是否同步更新？
  □ 改价 + 出库并发 → 按旧价出还是新价出？
  □ 报表中"销售金额"和"成本金额"口径是否一致？
```

**难点 2：并发出库（超卖风险）**

```text
库存 10 个，50 个用户同时下单出库
→ 只有前 10 个成功
→ 多一个都是超卖

这是线上最容易出事故的点——偶发的并发 Bug 平时抓不到，
一出就是财务问题。
```

---

### 三、库存模块难点：多仓库一致性 + 状态隔离

**难点 1：多仓库数据互相不串**

```text
□ WH001 的库存变化不能影响 WH002
□ 调拨：A 仓 -10、B 仓 +10 → 两仓总和不变
□ 两仓同时操作同一个 SKU → 锁是否正确隔离
```

**难点 2：库存状态的正确流转**

```text
可用库存 → 锁定（审核后）→ 已出库
         → 占用（调拨中）→ 已到达
         → 冻结（盘点中）→ 已解冻

测试难点：
  □ 锁定库存不能再次锁定
  □ 已出库的不能召回为可用
  □ 调拨中的库存同时被出库 → 锁冲突
  □ 盘点中改库存 → 盘点数据不准
```

---

### 四、三模块联动的核心难点（最致命）

这三个模块不是独立的——它们的联动是最难测的：

```
采购入库   → 库存 +100  → 应付凭证生成
销售出库   → 库存 -50   → 应收凭证生成
销售改价   → 库存成本更新 → 毛利重算
销售作废   → 库存 +50   → 凭证冲红    ← 三步必须全部回滚
调拨       → A仓-50 B仓+50 → 库存守恒

测试策略：
  □ 每个操作后跨模块做对账断言
  □ 偶发回调绑定全链路回归
```

---

### 五、面试回答模板

> **"三个模块的难点各不同，但核心都是数据一致性。采购难点是分批入库的状态流转和应付对账——分两批入库后凭证怎么生成、金额对不对。销售难点是改价后的成本同步和并发出库的超卖风险——改价时要验证库存成本和财务口径都跟着变了，并发时 50 人抢 10 个库存不能超卖。库存难点是多仓库数据隔离和状态流转——A 仓 B 仓不能串、调拨后两仓总和不变、锁定了不能再锁。**
>
> **最致命的是三模块联动——比如销售作废，库存没回滚、凭证没冲红，其中一个没做到就会导致月底对账不平。"**

---

**关键词**：进销存难点 / 采购入库 / 销售改价 / 库存状态 / 多模块联动 / 超卖

## Q22. 写测试用例的时候，你怎么保证用例覆盖是完整的？

### 问题描述

面试官问你：你写了 100 条用例，怎么证明该测的都测到了，没有遗漏？考的是你的**覆盖度保证体系**。

### 答案

用例覆盖不是凭感觉，我靠 **四层保障**：设计方法 → 业务维度 → 评审兜底 → 线上反馈补漏。

---

### 一、设计方法保证基础覆盖

```text
等价类 + 边界值 → 保证输入不遗漏
  例：出库数量 0/1/库存量/库存量+1/999999 → 5 条用例覆盖所有值域

场景法 → 保证业务流程完整
  例：创建订单 → 审核 → 出库 → 凭证生成 → 一条用例跑通全链路

错误推测法 → 保证异常场景不遗漏
  例：之前线上出现过"改价+并发出库导致成本不一致"
  → 这个场景必须加入用例库
```

---

### 二、业务维度保证分层完整

拿到一个功能，我习惯按 **7 个维度** 逐项检查：

```text
□ 正向：正常操作能不能走通？
  例：销售改价 100→150 → 成功

□ 反向：异常输入能不能拦截？
  例：改价输入负数 → 提示"价格不能为负"

□ 状态流转：状态之间切换对不对？
  例：已审核的订单不能再修改

□ 数据一致性：跨模块的数据对得上吗？
  例：改价后 → 库存成本同步更新 → 财务毛利重算

□ 并发：多人同时操作会不会冲突？
  例：50 人同时改同一商品价格

□ 幂等：重复操作会不会产生副作用？
  例：两次提交同一改价请求 → 价格只改一次

□ 回滚：操作撤销后数据有没有还原？
  例：改价后作废重改 → 最终价格正确
```

---

### 三、评审机制兜底

```text
自己写完 → 和开发一起评审用例
重点问三个问题：
  1. "这个异常分支你代码里处理了吗？"
  2. "用户如果连点两下提交，你怎么处理的？"
  3. "这个字段的边界值在哪？"

评审时经常发现我遗漏的并发场景和状态流转边界。
```

---

### 四、线上反馈持续补漏

```text
每出一个线上问题 → 回溯：对应的测试用例有没有？
  有 → 用例不够严密，优化
  没有 → 补一条用例，以后不会再漏

比如之前改价+并发出库导致成本不一致，
就是因为没测"改价时正在出库"这个场景。
上线出了之后，我立刻补了这条并发用例，并入回归库。
```

---

### 五、AI 大模型辅助生成用例（提效但必须 Review）

我现在会先用 AI 帮我出一版初稿，再用自己的经验去补漏：

```text
给 AI 的 Prompt：
"ERP 系统的销售改价功能，帮我生成测试用例。
  要求覆盖：
    1. 正常改价（调高/调低）
    2. 异常（负数、空值、超时）
    3. 边界（0、999999.99）
    4. 并发（50 人同时改）
    5. 数据一致性（改价后成本同步、毛利重算）
    6. 幂等（重复提交）
  输出格式：表格，含用例编号+场景描述+前置条件+操作+预期"
```

**AI 能帮我做的**：

```text
✅ 快速生成基础框架 —— 正向 3-4 条、异常 5-6 条，1 分钟出稿
✅ 补充我可能漏掉的常规维度 —— 比如边界值的极端值
✅ 输出格式统一 —— 省去排版时间
```

**AI 做不到的（必须我自己来）**：

```text
❌ 理解业务上下文 —— AI 不知道"改价时如果正在出库"会发生什么
❌ 知晓历史踩过的坑 —— AI 不知道之前线上出过什么问题
❌ 判断什么场景在生产环境最容易炸 —— 这是经验

所以 AI 出的用例我会做三件事：
  1. 补充业务独有的联动场景（改价+出库并发、改价+财务对账）
  2. 加入历史踩过的坑
  3. 删掉不相关的、调整不准确的描述
```

**实际效果**：以前一个功能从头写用例要 2 小时，现在 AI 出初稿 1 分钟，我花 30 分钟打磨补充，效率提升 60%。

---

### 六、面试回答模板（含 AI 辅助）

> **"我保证用例覆盖有四层。第一是靠设计方法——等价类边界值、场景法、错误推测法。第二是用 7 维度清单逐项检查每个功能。第三是用例评审和开发一起过。第四是线上反馈闭环补漏。**
>
> **另外我现在会先用 AI 出一版初稿——给 AI 设好维度和格式约束，让它快速生成基础框架。但业务的联动场景和历史踩坑必须我自己补，AI 替代不了对 ERP 业务的理解和经验。AI 出稿 1 分钟，我打磨 30 分钟，效率提升 60%。"**

---

**关键词**：测试用例覆盖 / 7维度 / 用例评审 / AI辅助生成 / 线上反馈

| | 共用？ | 为什么 |
|:---|:---:|:---|
| **关键字引擎** | ✅ 共用 | 执行逻辑都一样，不能每个模块写一版 |

```text
✅ 共用一套的好处：
  - 引擎改了全局生效，不用每个模块改一遍
  - 关键字库积累到 20 个左右，所有模块都能用
  - 新人上手学一套就行，新增模块只需要写 YAML 和用例
```

> **但 YAML 数据和测试用例是按模块独立的——tests/sale、tests/purchase、tests/wms 分目录管理，互不干扰。运行策略也能分开——PR 提交只测当前模块，全量回归才一起跑。"**

## Q25. 接口自动化主要覆盖和校验哪些东西？

### 问题描述

面试官问：你的接口自动化测了什么？不只是看 200 吧？

### 答案

接口自动化测的不只是"通了没"，我分了 **6 个校验维度**：

---

### 一、状态码校验（最基础）

```python
assert resp.status_code == 200   # 正常
assert resp.status_code == 401   # 未登录
assert resp.status_code == 422   # 业务校验失败（库存不足）
```

---

### 二、响应字段校验

```python
data = resp.json()

# 类型校验
assert isinstance(data["amount"], (int, float))

# 值校验
assert data["status"] == "PAID"
assert data["amount"] == 100.00

# 存在性校验
assert "token" in data

# 空值校验
assert data["error_msg"] is not None, "报错时必须有错误信息"

# 列表校验
assert len(data["items"]) > 0
```

---

### 三、业务逻辑校验

```python
# 审核后状态必须变
assert data["status"] != "PENDING"

# 库存不能为负
assert data["quantity"] >= 0

# 单据号唯一
order_ids = [r.json()["order_no"] for r in results]
assert len(order_ids) == len(set(order_ids)), "存在重复订单号"

# 金额精度——财务必须精确到分
assert round(data["amount"], 2) == data["amount"]
```

---

### 四、跨接口数据一致性（最核心）

```python
# 出库金额 vs 凭证金额逐笔对账
outbound_amount = outbound_resp.json()["amount"]
voucher_amount = voucher_resp.json()["amount"]
assert abs(outbound_amount - voucher_amount) < 0.01, \
    f"对账不平！出库={outbound_amount} 凭证={voucher_amount}"

# 调拨后库存守恒
assert stock_total_after == stock_total_before
```

---

### 五、异常容错

```python
# 参数缺失时返回 400 且有错误提示
resp = requests.post("/order/create", json={})  # 不传参数
assert resp.status_code == 400
assert "必填" in resp.text

# 异常数据不崩系统
resp = requests.post("/order/create", json={"quantity": -1})
assert resp.status_code in [400, 422]  # 不崩，有拦截
```

---

### 六、性能基线

```python
# 响应时间不能超过阈值
import time
start = time.time()
resp = requests.post("/outbound/audit", json={...})
elapsed = time.time() - start
assert elapsed < 3.0, f"出库审核超时: {elapsed:.2f}s"
```

---

### 校验维度速查

| 维度 | 校验什么 | ERP 场景 |
|:---|:---|:---|
| 状态码 | 200/400/401/422 | 未登录必须 401 |
| 字段 | 类型、值、存在性 | amount 必须是数字 |
| 业务逻辑 | 状态流转、精度、唯一性 | 审核后状态不能是 PENDING |
| **对账** | 跨接口数据对齐 | **出库 vs 凭证金额** |
| 容错 | 异常参数不崩 | 空 body 返回 400 |
| 性能 | 响应时间 < 阈值 | 出库审核 < 3s |

### 面试回答模板

> **"接口自动化我主要是检查6个东西。基础的状态码和字段校验——200、amount对不对、token有没有。业务逻辑校验——审核后状态变了没、库存不能为负。最核心的是跨接口数据一致性对账——出库金额和凭证金额逐笔比对，不是看接口通没通，是看数据对不对齐。还有异常参数容错和响应时间基线。"**

**关键词**：接口校验 / 对账 / 数据一致性 / 字段校验 / 性能基线

## Q26. 以"采购入库"接口为例，具体校验哪些东西？

### 问题描述

面试官追问具体接口的验证逻辑，考的是你能不能从通用方法论落到具体业务。

### 答案

以采购入库接口 `POST /purchase/inbound/audit` 为例，接口入参是 `{inbound_id}`，我会从 **7 个层面** 来校验：

---

### 1. 状态码

```python
resp = api_client.post("/purchase/inbound/audit", json={"inbound_id": 1001})
assert resp.status_code == 200   # 正常
assert fail_resp.status_code == 422  # 库存已满/单据已审核
```

---

### 2. 响应字段

```python
data = resp.json()
assert isinstance(data["inbound_id"], int)     # 字段类型
assert data["status"] == "AUDITED"             # 状态变更
assert data["warehouse_name"] is not None      # 仓库名有返回
assert len(data["items"]) > 0                  # 入库明细不为空
```

---

### 3. 业务逻辑

```python
# 入库数量必须 > 0
assert data["total_quantity"] > 0

# 入库后单据状态不能是 PENDING
assert data["status"] != "PENDING"

# 入库时间应在审核时间之后
from datetime import datetime
assert datetime.fromisoformat(data["inbound_at"]) > \
       datetime.fromisoformat(data["created_at"])

# 同一单据不能重复审核（幂等）
resp2 = api_client.post("/purchase/inbound/audit", json={"inbound_id": 1001})
assert resp2.status_code == 422
assert "已审核" in resp2.text
```

---

### 4. 数据一致性（跨接口对账——最关键）

```python
# 入库前查库存
stock_before = api_client.get("/wms/stock/query",
    params={"sku": "SKU001", "warehouse": "WH001"}).json()["quantity"]

# 入库审核
resp = api_client.post("/purchase/inbound/audit", json={"inbound_id": 1001})
inbound_qty = resp.json()["total_quantity"]

# 入库后查库存
stock_after = api_client.get("/wms/stock/query",
    params={"sku": "SKU001", "warehouse": "WH001"}).json()["quantity"]

# 对账：库存增加量 = 入库量
assert stock_after == stock_before + inbound_qty, \
    f"库存不一致！入库前={stock_before} 入库量={inbound_qty} 入库后={stock_after}"

# 应付凭证是否生成
voucher = api_client.get("/finance/voucher/query",
    params={"biz_type": "purchase_inbound", "biz_id": 1001}).json()
assert len(voucher["items"]) > 0, "入库后未生成应付凭证"
assert voucher["items"][0]["amount"] == resp.json()["total_amount"], \
    "凭证金额与入库金额不一致"
```

---

### 5. 异常参数容错

```python
# 入库 ID 不存在
resp = api_client.post("/purchase/inbound/audit", json={"inbound_id": 99999})
assert resp.status_code == 404

# 空 body
resp = api_client.post("/purchase/inbound/audit", json={})
assert resp.status_code == 400
assert "inbound_id" in resp.text  # 有明确错误提示

# 超长/非法 ID

```

### 6. 权限校验

```python
resp = requests.post("/purchase/inbound/audit", json={"inbound_id": 1001})
```

### 7. 性能基线

## Q27. 自动化是用于项目测试还是线上回归？

### 问题描述

面试官问你：自动化跑的这套东西，是在测试环境跑，还是线上跑？

### 答案

**两个环境都跑，但策略完全不同——测试环境全量，线上只跑读接口。**

---

### 测试环境：全量跑

```text
什么时候跑：
  - PR 提交 → 跑冒烟 + 单接口
  - 版本上线前 → 跑全量链路 + 对账
  - 每日凌晨 → 定时回归全量

跑什么：
  ✅ 全量——读接口、写接口、业务链路、对账
  ✅ 脏数据无所谓——跑完自动清理（fixture yield 自清洁）
```

---

### 线上环境：只跑读 + 拨测

```text
什么时候跑：
  - 每 5 分钟 → 拨测核心接口
  - 版本上线后 → 灰度回归

跑什么：
  ✅ 读接口：库存查询、订单查询、凭证查询
  ❌ 写接口绝不对上——不能在线上创建测试订单！
  ✅ 数据一致性巡检：扫异常数据（负数库存、已出库无凭证）

写操作怎么办？
  不在线上自动化里做。需要线上验证用灰度环境或仿真环境。
```

---

### 为什么线上不能跑写接口

```text
❌ 线上创建测试订单 → 污染生产数据
❌ 线上调出库审核 → 真实库存被扣
❌ 线上跑并发压测 → 影响真实用户

✅ 线上的岗位：拨测读接口（查库存/查订单）+ 巡检（扫异常数据）
  写操作只走测试环境和预发环境
```

---

### 面覆回答模板

> **"两个环境都跑，但分工完全不同。测试环境跑全量——读接口、写接口、业务链路、对账都测，有 fixture 自清洁不怕脏数据。线上只跑读接口的拨测——每 5 分钟探一下库存查询、订单查询有没有挂，外加数据一致性巡检扫异常数据。写操作绝不对线——不能在生产库建测试订单。"**

---

**关键词**：测试环境 / 线上回归 / 线上拨测 / 读写分离

## Q29. 冒烟测试的 scope 会随着版本迭代扩大吗？

### 问题描述

面试官问冒烟测试的覆盖范围是否会随版本迭代扩大——这个问题考的是你对**测试分层策略**的理解。冒烟的本质是"快速验证主链路没挂"，如果 scope 无限扩大，就等于没有冒烟了。

---

### 答案

#### 一句话核心

> **冒烟 scope 要动态调整，但原则是"最小可用"验证，不能越跑越长。**

---

### 一、核心原则

```
冒烟 ≠ 全量回归
冒烟 = 核心链路没挂，可以继续测
```

冒烟的理想耗时：**5-10 分钟**
超过 15 分钟 → 说明 scope 膨胀了，需要瘦身

---

### 二、动态调整的三条规则

**① 新功能上线初期 → 加进冒烟**

刚上的功能加进冒烟，跑基本正向流程，确保没有把主干打挂。

```
例：新上线"库存调拨"功能
冒烟加一条：调拨单创建 → 审核 → 目标仓库库存+1
等这个模块稳定（跑过2-3个版本没出大问题），
就从冒烟里移出去，交给全量回归覆盖。
```

**② 核心链路 → 永远留在冒烟里**

```
采购入库 → 审核 → 库存+1
销售出库 → 审核 → 库存-1，资金记录生成
登录 → 权限正确
```
这几个是系统能不能用的底线，**不管怎么迭代都留在冒烟里**。

**③ 接口冒烟和 UI 冒烟分开管理**

| 类型 | 触发时机 | 耗时 | 覆盖内容 |
|------|-----------|------|----------|
| 接口冒烟 | 每次提测 | 2-3分钟 | 核心接口链路正向流程 |
| UI冒烟 | 发版前 | 5-8分钟 | 核心页面加载+关键操作 |

---

### 三、结合你的项目回答（面试话术）

```
冒烟的scope我会随着版本迭代动态调整，
但有一个原则：冒烟永远是"最小可用"验证，
不能越跑越长。

具体分三种情况：

第一，新功能上线初期我会加进冒烟，
跑个基本的正向流程，
确保没有把主干打挂。
等这个模块稳定了，就从冒烟里移出去，
交给全量回归覆盖。

第二，核心链路永远在冒烟里——
采购入库、销售出库、库存查询，
这几个是系统能不能用的底线，
不管怎么迭代都留在冒烟里。

第三，接口冒烟和UI冒烟分开——
接口冒烟跑得快，每次提测都跑；
UI冒烟主要在发版前跑。

目前我们的冒烟用例大概是
接口20条左右 + UI 8-10条核心路径，
控制在10分钟以内跑完。
```

---

| 说法 | 为什么加分 |
|------|-------------|
| "冒烟控制在10分钟以内" | 说明你有量化意识 |
| "稳定后从冒烟移出去" | 说明你理解冒烟和回归的区别 |
| "接口和UI冒烟分开触发" | 说明你有工程化思维 |
| "核心是最小可用验证" | 直达问题本质 |


## Q41. 支付成功但订单仍显示"未支付"，可能的原因是什么？

### 问题描述

用户在电商平台完成支付，第三方支付平台（微信/支付宝/银联）已扣款成功，但回到订单页面，订单状态仍然是"未支付"。作为测试工程师，请从全链路角度分析可能的原因，并说明排查思路。

> 这是一道高频的**场景设计+系统理解**面试题，重点考察对支付系统全链路的理解，以及问题排查的结构化思维。

---

### 全链路分析

支付系统涉及**用户端 → 商户系统 → 第三方支付平台 → 银行**多个环节，任何一个环节出问题，都可能导致状态不一致。

整体链路：

```
用户支付
  → 第三方支付平台扣款
    → 异步回调（callback/notify）通知商户系统
      → 商户系统更新订单状态
        → 前端展示最新状态
```

任何一个箭头出问题，都会导致"已扣款但订单未支付"。

---

### 一、第三方支付平台侧问题

#### 1. 回调通知未发出或丢失

- 用户支付成功，但支付平台**异步回调没有发出**
- 网络抖动导致回调丢包
- 支付平台自身Bug（罕见，但偶有发生）

#### 2. 回调地址配置错误 ⭐ 高频

- 商户平台配置的回调URL有误
- **环境搞错**：沙箱环境配置了生产回调地址，或反之
- 回调接口返回非200状态码，支付平台认为通知失败，进入重试队列

---

### 二、回调接收与处理环节

#### 3. 回调接口本身故障

- 回调接口服务宕机/重启，请求到达时服务不可用
- 回调接口超时（支付平台有重试机制，但重试次数有限，一般3-5次）

#### 4. 签名验证失败 ⭐⭐ 最高频坑

- 回调请求到了，但**签名校验失败** → 系统主动拒绝处理
- 常见原因：
  - 密钥（app_secret）配置错误（测试环境/生产环境用混）
  - 编码不一致（UTF-8 vs GBK）
  - 参数排序规则不同（字母序 vs 字典序）
  - 误把sign字段也参与签名计算

#### 5. 回调处理的幂等性问题

- 同一笔支付回调发了多次（支付平台重试机制）
- 系统没有做幂等处理，或幂等逻辑有Bug
- 导致：重复处理时状态异常，或唯一键冲突抛异常

---

### 三、订单状态更新环节

#### 6. 数据库事务未提交或回滚 ⭐

- 回调处理过程中，更新订单状态的SQL执行了，但**事务未提交**
- 或者后续逻辑异常导致事务回滚
- 现象：日志显示"更新成功"，但数据库里状态没变

#### 7. 消息队列延迟/堆积 ⭐⭐ 高频

- 回调接收后，订单状态更新是**异步处理**（扔到MQ）
- MQ消费慢、堆积，导致状态更新严重延迟
- 用户刷新页面时还没消费到，看到"未支付"
- **这是最容易被忽视的原因**

#### 8. 订单状态机设计问题

- 订单状态流转逻辑有Bug
- 例如：只允许 `待支付 → 已支付`，但某些条件下状态被锁住
- 并发情况下两个请求同时更新，后者覆盖前者（丢失更新）

---

### 四、前端展示环节

#### 9. 前端缓存未刷新

- 订单详情页做了本地缓存，支付完成后没有主动刷新
- 需要用户手动下拉刷新或重新进入页面才能看到最新状态

#### 10. 前端轮询/主动查询逻辑缺失

- 支付完成后，前端没有主动调用"查询订单状态"接口
- 完全依赖后端推送（而推送可能失败）
- **正确做法**：支付完成后，前端应定时轮询订单状态，或引导用户手动刷新

---

### 五、超时与边界问题

#### 11. 用户支付时网络中断

- 用户发起支付，支付平台扣款成功，但**同步返回结果**时网络断了
- 用户端以为支付失败，实际上已扣款
- 订单状态可能还是"未支付"（等待异步回调）

#### 12. 超时关单与支付完成的时序冲突 ⭐

- 系统设置了"支付超时自动关单"（如30分钟）
- 用户在超时边缘完成支付，回调到达时订单已被关单处理逻辑锁定
- 导致：支付成功，但订单被系统自动关闭（退款流程触发）

---

### 六、作为测试工程师的排查思路

```
第一步：确认支付平台是否真的扣款成功
  → 登录商户平台，查交易流水
  → 确认支付平台侧状态

第二步：查回调日志（最关键）
  → 回调请求是否到达我方系统？
  → 签名验证是否通过？
  → 返回给支付平台的状态码是什么？
  → 处理过程中有无异常日志？

第三步：查订单状态变更日志
  → 数据库里订单状态是什么？
  → 状态变更的记录有没有？时间点是什么？
  → 有没有事务未提交的迹象？

第四步：查MQ消费情况（如果有异步处理）
  → MQ消息是否发送成功？
  → 消费者是否消费？有无异常？
  → 消息是否堆积？

第五步：查前端逻辑
  → 支付完成后的跳转逻辑是否正确？
  → 有没有主动查询订单状态？
  → 本地缓存是否清除？

第六步：复现验证
  → 在测试环境用Mock模拟各种失败场景
  → 验证系统的补偿机制是否生效
```

### 八、延伸：系统的补偿机制设计

面试官可能追问："你们系统是怎么解决这个问题的？"

**标准答案参考**：

```
1. 被动补偿（支付平台重试）
   - 回调接口返回非200时，支付平台会重试（一般3-5次）
   - 我方系统要保证幂等，重复回调不报错

2. 主动补偿（定时任务扫描）
   - 每5分钟扫描"待支付"超过10分钟的订单
   - 调用支付平台查询接口，确认实际支付状态
   - 如果已支付，自动更新订单状态

3. 用户侧补偿
   - 订单列表/详情页，提供"重新查询"按钮
   - 用户点击后，调用查询接口刷新状态

4. 人工补偿（兜底）
   - 客服系统支持"补单"操作
   - 根据支付流水号，手动更新订单状态
```

---

**关键词**：支付回调 / 异步通知 / 签名验证 / MQ延迟 / 事务回滚 / 补偿机制 / 幂等设计 / 支付状态一致性
