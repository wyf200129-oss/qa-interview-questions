# 性能测试面试题

---

## Q1. JMeter 压测怎么用？TPS、响应时间怎么看？

### 问题描述

面试官问你用没用过 JMeter 做性能测试，不是在问你装没装过，而是问你**知不知道怎么看指标、怎么定标准**。

### 答案

做 ERP 项目时用 JMeter 压过出库审核和凭证生成接口。核心操作三步走 + 三个必看指标。

---

### 一、JMeter 核心操作三步

```text
Step 1：创建线程组
  线程数：50（模拟 50 个用户）
  Ramp-Up：5s（5 秒内启动完 50 个线程）
  循环次数：10（每人操作 10 次）→ 总请求 500 次

Step 2：添加 HTTP 请求
  协议：https
  IP：test-api.example.com
  方法：POST
  路径：/wms/outbound/audit
  Body：{"order_id": "xxx"}

Step 3：添加监听器看结果
  聚合报告：看 TPS / 响应时间 / 错误率
  查看结果树：看单个请求的成功/失败详情
  用表格查看结果：看每个请求的具体耗时
```

### 二、三个必看指标

| 指标 | 含义 | ERP 标 | 怎么看 |
|:---|:---|:---|:---|
| **TPS** | 每秒处理请求数 | > 100 | 聚合报告 Throughput |
| **平均响应时间** | 平均多久返回 | < 500ms | 聚合报告 Average |
| **95% Line** | 95%的请求在多久内完成 | < 1000ms | 聚合报告 95% Line |

### 三、聚合报告怎么看

```
Label           #Samples  Average   Median    95% Line    Min    Max    Error%   Throughput
出库审核接口      500       320       280       850        120    5200    0.20%    45.2/sec
```

解读：
- 500 次请求，平均 320ms，95% 在 850ms 内完成 ✅
- 最大 5200ms → 有慢请求，可能是数据库锁 ✅ 关注
- 错误率 0.2% → 500 次中有 1 次失败 → 查一下是不是并发冲突 ⚠️
- TPS 45/秒 → 一秒处理 45 个出库审核

### 四、参数化：登录后拿 token 传给后续请求

```text
1. 线程组 → 加 HTTP Request (#1 登录)
2. 这个请求下面加 JSON Extractor 提取 token
   Names of created variables: token
   JSON Path expressions: $.data.token

3. 第二个 HTTP Request（出库审核）的 Header 加
   Authorization: Bearer ${token}
```

### 五、ERP 压测场景

```text
出库审核压测：
  50 用户并发，每人审核 10 次
  目标 TPS > 50
  目标 95% 响应 < 1s
  目标错误率 < 1%

如果结果不达标：
  查数据库慢 SQL → 加索引
  查有没有锁冲突 → 加乐观锁
  查服务器 CPU/内存 → 扩容
```

### 面试回答模板

> **"我用 JMeter 做过出库审核接口的压测。50 线程并发，每人审核 10 次，总 500 次。看聚合报告里的 TPS、平均响应时间、95% Line 和错误率。我们定的是 TPS > 50、95% 响应 < 1s、错误率 < 1%。**
>
> **如果指标不达标，我会配合服务器日志和数据库慢查询来定位瓶颈——是数据库锁问题还是代码没加缓存。"**
